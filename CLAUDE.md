## Контекст

Строим автономного AI-агента продаж для детского центра «Мама рядом» (Mama Ryadom), Уфа.
Бот ведёт клиента от заявки до зачисления, **обрамляя КЭВ** (см. ниже): общается в
мессенджерах, помнит контекст между сессиями, ведёт карточку в CRM, ставит задачи,
бронирует/перезаписывает пробные, анализирует записи звонков продавца.

- **Ниша:** B2C, детский центр, дети **1,5–4 года** (черновик заказчика; по факту — по нескольким критериям). Три направления —
  **гимнастика** («Мама и малыш»), **беби балет**, **сенсорика**; детали в разделе «Продукты и возрастные критерии». Цикл сделки длинный и рваный (заявка → дозвон со
  2–3 раза → молчит днями → просит позвонить → ушёл подумать → записался → не пришёл →
  перезапись → уехал на 2 недели → и только потом купил). Это закладываем в память и задачи.
- **Команда:** один продавец. Маршрутизация/распределение заявок не нужны.
- **Масштаб:** пока один бизнес («Мама рядом»). Мультитенант не закладываем.
- **Backend/CRM:** **Fitbase** (`https://api.fitbase.io/api/v2`). Не Firebase — это была
  опечатка. Старая CRM `lk.ufagymnastics.ru` выводится из эксплуатации (см.
  `old-bot-architecture.md` как легаси-референс).

**Подход к автономии:** цель — полная автономия, но строим поэтапно через
**human-in-the-loop**: бот действует/предлагает, продавец может вмешаться. «Страховочные
колёса» снимаем по мере роста доверия. Каркас под автономию ≈ +20% к версии «помощник»
(не «с нуля автономного»). Полную автономию НЕ делаем условием запуска.

## Роль бота — «обрамляет КЭВ»

**КЭВ** (ключевой этап воронки) = **пробная тренировка с презентацией** — момент, где
продаётся основной продукт (абонемент).

**Пробная — средство, не обязательный барьер.** Цель — продать абонемент; пробная лишь
помогает. Готов лид оформить абонемент **сразу, без пробной** — бот продаёт сразу, не гонит
через пробную. Редко, но бот не должен «заставлять» проходить пробное.

- **До КЭВа — целиком бот:** реакция на заявку ≤5 мин, квалификация, «продать» пробную и
  записать. Здесь главная зона автономии.
- **КЭВ — оффлайн, человек:** саму пробную ведёт тренер/продавец. Бот не внутри.
- **После КЭВа — снова бот:** подтверждение прихода, догрев «пришёл → думает → зачислен»,
  обработка неприхода и перезаписи.

## Зона бота — только продажи (лиды), не действующие клиенты

Бот работает **только с лидами / потенциальными клиентами**. С действующими (ходящими)
клиентами он не общается и не выполняет по ним действий — это отдельная база, **зона админа**.

**Роли (для ясности терминологии):** действующие (платящие, ходящие) клиенты — забота
**админа**; заявки и те, кто ещё не оплатил абонемент — забота **продавца**. Бот **занимает
роль продавца** (лиды / неоплатившие).

- На входе в диалог бот **проверяет надёжные идентификаторы** (телефон / id канала: MAX, VK,
  TG) по базе действующих клиентов Fitbase (напр. активный `ClientContract`).
- Совпало по надёжному ключу → действующий клиент: чат уходит **админу**, бот **отключается**
  в этом чате (не тратит ресурсы). **По имени НЕ сверяем** (решение 2026-05-29): тёзок слишком
  много («куча Гузелей»/Елен), ложное отключение лида дороже редкого случая «клиент написал с
  нового номера» — такого бот спокойно ведёт как лида, без тихих заметок админу.
- Когда лид **становится** клиентом (купил абонемент): на подтверждённой оплате — **handoff
  продавцу** (задача+алерт «проверь оплату, дальше веди сам»). Это граница лид→клиент. Продажу
  (дожим до покупки) ведёт бот (F6); пост-продажный онбординг (связать с тренером, «теперь
  вопросы сюда») = **F7**, пока вручную человеком.
- **Бывший клиент вернулся** (абонемент кончился) — снова **лид** (бот прогревает), но
  карточка/память помнят, что уже ходили. Кейсов мало.

## Каналы (транспорт)

| Канал | Доля трафика | Интеграция | Когда |
|---|---|---|---|
| **MAX** | ~60% | Wazzup (проверить, что отдаёт MAX-канал; иначе MAX API напрямую) | **Старт** |
| WhatsApp | ~30% | Wazzup (QR-привязка — окна 24ч нет) | Фаза 1/2 |
| Telegram | ~5% | Wazzup / Bot API | Позже |
| VK | ~5% | Wazzup / VK API | Позже |
| Телефон | — | Мегафон-АТС (только запись → анализ) | Фаза 2 |

- **Старт с MAX** — самый большой объём = максимум реальных диалогов для итераций.
- Логика агента **не зависит от канала** (роутер). Fitbase Lead хранит идентификаторы
  каналов в полях (`phone, max, vk, telegram, instagram, email`) → один человек = один лид,
  мультиканальная дедупликация на уровне модели данных (+ поле `is_duplicate`).
- **Модель — все каналы сразу** (опознание не зависит от канала, якорь — телефон; чат без
  телефона авто-сшить можем не суметь — край на потом); **транспорт — по мере включения**
  канала (MAX старт, дальше по «Когда» в таблице).
- **Бот не звонит сам.** Звонит продавец, звонок записывается, запись анализируется.
- **VK — два разных входа:** чат сообщества ВК = канал двери A (строка «VK» выше); **VK Ads
  (таргет) лид-формы** = источник двери B (близнец Тильды), в личку не пишем. См. «Две двери
  входящего».

### Что где живёт

**Fitbase — источник правды** (всё через REST, Bearer + заголовок `domain`):

| Нужное боту | Сущность Fitbase |
|---|---|
| Карточка клиента / лид (+ каналы, источник, теги, `custom_fields`) | `Lead`, `Client`, `Contact` |
| Этап воронки | `Lead.funnel_step` (+ `PurchaseFunnel/PurchaseFunnelStep`) |
| Источник заявки | `advertising_source` |
| Что продано + факт оплаты | `ClientContract.payment/payment_date`, `ClientService.pay_date/done` |
| Запись на пробную / перезапись | `ScheduleRegistration` (+ `auto_delete_time`) |
| Приход на пробную | `ClientVisits` |
| Задачи продавцу | `Task` (`responsible`, `available_to`, `description`, `result`) |
| Заметки (видны продавцу) | `Note` |
| Сотрудники | `User` |

**Наш тонкий слой** (то, чего нет в Fitbase): транскрипты диалогов · долгосрочная и
эпизодическая память + ретривал · состояние агента · планировщик проактивных касаний ·
цикл опроса Fitbase.

**Вебхуков для своей интеграции в Fitbase нет** → состояние Fitbase читаем **опросом
(polling)**, инкрементально по `updated_at` («что изменилось с прошлой проверки»).
Мгновенное (сообщение клиента) приходит push-ом из канала, не через опрос. Подтверждено
разрабами Fitbase (2026-05-27): вебхуки есть только в их готовых интеграциях (Тильда,
Клауд), нам через REST их не дают. **Rate limit — 20 rps на API** (40 / burst 80 — общий,
не наш потолок); для нашего масштаба запас огромный. Реальный риск — пробить лимит своим
багом (цикл-повтор / частый опрос) → все вызовы Fitbase идут через один «турникет»: мягкий
темп + бэк-офф на 429 («слишком часто»).

## Две двери входящего (A — клиент написал · B — заявка в CRM)

Входящее бывает двух видов с разными триггерами. Схема выше показывает только дверь A —
дверь B так же важна (а по объёму, возможно, и основной поток лидов).

**Дверь A — клиент написал сам.** Сообщение в MAX / WhatsApp / чат сообщества ВК / … → канал
толкает push. Тред уже есть, отвечаем в него.

**Дверь B — заявка появилась в CRM, человек ещё не писал.** Лид родился в Fitbase из:
- **Тильда-форма** (лендинг наш → можем слать заявку и нам вторым адресом — мгновенный детект);
- **VK Ads (таргет), лид-форма** — по сути близнец Тильды: даёт телефон, в личку ВК **не**
  пишем (не путать с чатом сообщества ВК — это канал двери A);
- **руками продавцом** — входящий звонок вместо заявки, написали в Инсту.

На руках телефон (+ имя), но **не** мессенджер клиента и **нет** треда → бот должен сам выйти
на контакт.

**Как узнаём про дверь B:** Fitbase нам не пушит → ловим **опросом** (радар по новым лидам);
где можем, источник шлёт заявку и **нам напрямую** (Тильда) — тогда мгновенно, без задержки
опроса.

**Первое касание — каскад (бот делает САМ, не спихивает продавцу):**

```
заявка → бот пишет в MAX  (Wazzup даёт писать первым — подтв.)
   └ молчат ~2ч → WhatsApp
        └ молчат ~2ч → задача продавцу «позвони»
```

Каскад привязан к телефону; порядок каналов — **настройка** (под источник свой), «~2ч» —
параметр. Задача продавцу — **последний шаг, не дефолт**: спихнуть на продавца = сегодняшняя
ручная работа = бот не нужен. Смысл бота — выйти на контакт мгновенно и сам.

**Когда строим дверь B (уточнение сборки, 2026-05-29):** *детект + опознание + дедуп* лида двери B
— в **Блоке 0** (трубы). Сам *исходящий каскад* (бот пишет первым) — с **F1**, когда бот умеет
говорить; до F1 лид двери B детектится и паркуется задачей продавцу (осознанно временно). WhatsApp
(2-й шаг каскада) — тоже не в 1-й волне (он Фаза 1/2).

**Один человек = один лид.** Один и тот же человек может прийти в обе двери (заполнил форму
**и** написал в MAX). Роутер на каждое входящее проверяет «знаем ли уже?» (телефон / имя /
id канала) и цепляет к существующей карточке, не плодит дубль. Это фундамент памяти, которая
ходит за человеком по каналам (см. «Память клиента»).

## Воронка и этапы

Этапы = шаги воронки Fitbase (`funnel_step`):

`Неразобранное` → `В работе` → `Пробная (записан)` → `Пришёл на пробную` →
`Зачислен` / `Отказ`; сбоку `Резерв` (младше по возрасту / нет места) и
`Архив` (неквал, фейк-номер).

Все источники падают в «Неразобранное»: звонок (Мегафон-интеграция / руками продавцом),
сайт, WhatsApp, VK, Telegram, MAX.

«Пришёл на пробную» — отдельный этап намеренно: тут клиент часто долго думает/сомневается,
и это зона догрева бота.

## Продукты и возрастные критерии

> **Статус: черновик от заказчика (2026-05-28), окончательно не утверждён.** Записан, чтобы строить
> F1; цифры/правила заказчик подтвердит. Гимнастики ХГ/РГ нет — «Мама рядом» отдельный бизнес.

**Единая философия:** движение · музыка · ощущения — мягкое развитие малыша рядом с мамой.
**Три направления:**

1. **Гимнастика** (мягкая/развивающая) — это и есть продукт **«Мама и малыш»**. Полосы препятствий,
   мягкие модули, тоннели, баланс, прыжки, лазание, координация, мячи, ловля, перекаты, крупная
   моторика, ориентирование в пространстве.
2. **Беби балет** (новый) — ритмика: учится танцевать и слышать музыку. Классическая музыка, чувство
   ритма, музыкальные паузы, мягкая растяжка, пластика, лёгкая хореография, осанка, координация.
3. **Мир ощущений / сенсорика** (новый) — учатся понимать мокрое/холодное/горячее, мелкая моторика.
   Вода, песок, крупы, желе, лёд, текстуры, пересыпания/переливания, пинцеты, пальчиковые игры,
   простые слова, повторения, звукоподражания, песни, совместные игры.

**Возраст и формат (все направления):**
- Возраст **1,5–4 года**; программы по уровням: **1,5–3** и **3–4 включительно**.
- Пол: м/ж; **балет — только девочки**.
- С мамой: **1,5–3 — только с мамой**; 3–4 — опционально.
- Вместимость: **до 6 пар в группе**, держим **≤5 действующих**, чтобы 1 место всегда было под пробную (иначе → резерв).

**Края (квалификация по возрасту):**
- **Младше 1,5** — взять можно, если: уверенно стоит/ходит сам без помощи; не боится новых людей
  (тренер, дети, родители), комфортно; коммуникация ок и готов к группе; мама готова. Учесть: **нет
  пеленальных столов и места для кормления**. Не проходит → **резерв** + напоминание написать, когда малышу ~1,4.
- **5+ интересуются** — жёстких ограничений нет: «прийти можно, но программа рассчитана примерно до 4
  включительно — в 5 может быть просто неинтересно».

## Память клиента

- **Краткосрочная:** текущий диалог (транскрипт).
- **Долгосрочная:** портрет — имя мамы/папы и **отдельно** имя/возраст ребёнка (бот различает их
  в диалоге и **обращается к ребёнку по имени**, когда узнал), программа/группа, договорённости,
  возражения, ограничения по расписанию.
- **Эпизодическая:** события — записался, не пришёл, перезапись, «звонок 12.05 сказал X»,
  «уехали на 2 недели», ключевые решения.

**Память = ретривал, а не только хранение:** в контекст модели кладём только релевантное,
не весь архив (иначе токены и стоимость взрываются — как в старом боте, где помог prompt
caching). Ключевые факты дублируем в Fitbase `Note`/`custom_fields`, чтобы их видел продавец.

**Память за человеком, не за каналом.** Контекст привязан к лиду; канал — лишь «дверь, через
которую сейчас говорят». Начали в MAX, продолжили в WhatsApp, вернулись в MAX — бот везде
помнит всё. Возможно это потому, что карточка одна (см. «один человек = один лид» в «Две двери
входящего»), а first-контакт инициируем мы — значит сами знаем, что разные треды = один человек.

**Активный канал.** Ответил где-то → этот канал становится активным, запланированная досылка в
другие гасится (не долбим в WhatsApp, если ответили в MAX). Активный канал — **настройка**,
можно перевести.

**Читать карточку перед ответом (источник правды — Fitbase).** Перед каждым ответом бот
подтягивает шапку карточки — этап воронки + ключевые поля + свежие заметки, — чтобы не
действовать на устаревшем (продавец только что записал на пробную / взял оплату / двинул этап).
Запрос мелкий, лимит Fitbase не трогает; кэш карточки держим **только как запаску** на случай
недоступности Fitbase. Это отдельно от опроса-радара: радар замечает изменения, пока клиент
молчит; чтение-перед-ответом — страховка в момент, когда клиент написал.

**Контекст звонка продавца → боту.** Пока: после звонка продавец пишет короткую заметку (о чём
говорили, что планирует) — бот читает её при подхвате диалога. Потом: автоматом из расшифровки
записи звонка (F9).

## Пробное и оплата

- Пробное **900₽** (черновик; было 500), засчитывается в стоимость абонемента.
- **Бронь = `ScheduleRegistration` в статусе ожидания + `auto_delete_time`** (дедлайн
  оплаты). Оплатил в окно → подтверждаем; не оплатил → бронь авто-снимается, слот свободен.
  Бот предупреждает клиента до истечения. Параметр «окно оплаты» (сколько часов) — задать.
- **Подтверждение оплаты, два пути:**
  - (а) эквайринг (вероятно **CloudPayments**) → факт оплаты бот узнаёт мгновенно: у CloudPayments
    есть **push-вебхук `Pay`** (`Amount`, `TransactionId`, `InvoiceId`, статус), в отличие от Fitbase
    (там вебхуков для нашей интеграции нет). Это **плановый конечный режим F3**; оба пути (человек +
    авто) собираем на F3. Открыто (решить на F3): аккаунт CloudPayments — **наш** (заказчик уверен)
    → вешаем свой notification-URL на `Pay`; сопоставление платежа с лидом (`InvoiceId` / связка
    Fitbase↔CloudPayments — вероятно подтягивает оплату в карточку, но не гарантировано); проверка
    подписи вебхука.
  - (б) пока — перевод + скрин в чат → бот фиксирует скрин и **эскалирует продавцу на
    проверку** (тот подтверждает оплату и разрешает продолжить); автодетект — позже.
- **Акции и скидки:** **редактируемый конфиг** (данные, не код) — список **проверенных** акций
  и скидок с прописанными правилами (напр. «не суммируются»). Скидку **из конфига бот применяет
  сам** — в т.ч. **проставляет поле `discount` в карточке Fitbase** (только значения из конфига,
  отсебятины нет); вне списка / спец-условия → эскалация. Заказчик меняет без релиза.
- **Прайс (черновик, по любому направлению):** разовое групповое **1300₽**. Абонемент на 1
  направление: 4 зан./1 мес — **5200**, 8 зан./2 мес — **8800**, 12 зан./2 мес — **12000** (занятие
  45 мин). **Единый абонемент** (2–3 направления, зависит от пола — у мальчиков нет балета): 8 зан. —
  **10500**, 12 зан. — **14000** (2 мес). Индивидуальные: разовое **2500**, 4 зан./1 мес — **9000**,
  8 зан./2 мес — **16000**. Скидка **−15% при двоих детях**.

## Эскалация / handoff (человек ↔ бот)

**Два смысла «передачи человеку» — не путать.** (1) **Граница бота** — штатная передача:
докуда бот доделан, там спокойно отдаёт эстафету продавцу (с саммари); граница **сползает
вниз по воронке** с каждым новым F-блоком. (2) **Эскалация** (ниже) — тревожный предохранитель:
«нужен человек прямо сейчас» на любом этапе, не двигается.

**Эскалация** = бот не решает сам и привлекает продавца. **Три режима** (для разных
триггеров — разные; точную карту «триггер → режим» делаем отдельной итерацией):
1. **Тихий handoff** — бот замолкает, диалог берёт продавец.
2. **Уведомление + бот продолжает** — спорный вопрос «паркуется» на человека: бот говорит
   клиенту «уточню у коллеги, вернусь» и продолжает вести диалог по остальному. **Алерт
   продавцу — в естественной точке, не в момент триггера** (иначе продавец влетит и порвёт
   горячий диалог): бот паркует непринципиальный вопрос, ведёт продажу дальше (подобрать время,
   записать), а задачу+алерт ставит, когда диалог дозрел (записались/оплатили) ИЛИ вопрос стал
   блокером ИЛИ лид замолк — по таймеру, чтобы не потерялось. Если непонятка может изменить саму
   рекомендацию (напр. здоровье) — к оплате не гнать. Срочное (агрессия, деньги, скрин, «хочу
   человека») — мгновенно, это уже тихий handoff. Обещание **обязательно подкрепляется** задачей.
3. **Только задача** — бот не прерывается, ставит задачу на будущее.

**Механика:** `Task` в Fitbase (`responsible`=продавец, `available_to`=срок,
`description`=контекст) + при срочности — **мгновенный алерт в Telegram**: кнопкой с опознавалкой
(`#4232 Маша · оплата`) + контекст (кто/почему/последняя реплика); тап → действия (`#старт` и пр.)
с подтверждением да/нет от случайного нажатия; несколько открытых — список, гасятся по одной.
Один продавец → без маршрутизации. Тот же механизм = **клапан автономии** (по мере доверия триггеры снимаем).

| Повод | Режим | Примечание |
|---|---|---|
| Скидка / спец-условия вне списка | Уведомление + продолжить | Стандартные скидки из списка бот даёт сам (см. «Пробное и оплата») |
| Здоровье / противопоказания | Уведомление + продолжить | На **продавца** |
| Возврат / спор по деньгам | Тихий handoff | Сначала «спасаем» (см. ниже), потом зовём; бот: «я уже позвал человека» |
| Конфликт / недовольство (в т.ч. после пробной) | Тихий handoff | Мгновенный алерт |
| Агрессия / «ухожу» | Тихий handoff | Мгновенный алерт |
| «Хочу человека» | Мягкий | Бот сначала предлагает помочь сам («закрою ~90% вопросов, чтобы не ждать»), при настойчивости — handoff |
| «Перезвоните» | Только задача | Задача-звонок + алерт |
| Не понял / не уверен / вопрос вне сценария | Тихий handoff | См. «Калибровку»: пытаться, но **в меру**, не блефовать |
| Скрин оплаты | Тихий handoff на проверку | Продавец проверяет и **разрешает продолжить** (пока оплату подтверждает человек) |
| Написал действующий клиент | Тихий handoff + бот off | См. «Зона бота» |
| Правка в CRM, которую бот не умеет | Только задача | Список «чего бот не умеет» — добить |
| Резерв / Архив | Подтверждение человеком | Бот предлагает — человек подтверждает |

**Возврат пробной — сначала спасаем, потом возвращаем.** Перед handoff бот уточняет причину:
«совсем неактуально — или просто сейчас неудобно?». Если ребёнок заболел / не успевают на
этой неделе — предлагает **перенос записи**, а не возврат (возврат по закону = заявление +
издержки на переводы; перенос удобнее всем). Возврат — крайняя мера.

**Неприходы — без жёсткого лимита.** Бот перезаписывает сколько угодно, неоплаченная бронь
авто-снимается (`auto_delete_time`). Даже «оплатил → вернули → снова хочет» — пожалуйста.

**Калибровка «когда сдаться»** *(добить)*: задать лимит попыток, прежде чем эскалировать (не
«переспросить 10 раз»), и явно разрешить боту признавать «не знаю, подключу человека» вместо
того чтобы быть полезным до последнего и блефовать.

**Ручной режим:** когда подключился человек — бот замолкает в диалоге **по этому лиду** (не весь
бот: с остальными лидами работает; состояние «ручной/авто» хранится по каждому лиду отдельно).
**Автоглушение:** ловим `isEcho`/чужой `authorId` в вебхуке Wazzup — продавец просто начал писать
клиенту, бот молчит. **Возврат:** продавец возвращает явной командой `#старт` (вводит там, где
клиент не видит — в Telegram-алерте, не в клиентском чате; пара: `#стоп`/автоглушение увести ↔
`#старт` вернуть) + авто-возврат после ~суток тишины, если ничего не висит — **кроме чувствительных
handoff (конфликт / деньги / агрессия): там только ручной `#старт`, авто-возврата нет.** Пока
молчит, бот всё равно записывает сообщения (без ответов); на `#старт` читает свежую карточку +
последние реплики и продолжает с контекстом.

## Дорожная карта (блоки)

Строим **поблочно**: каждый блок — автономная единица, снимающая кусок нагрузки с продавца.
Каждый стартует **human-in-the-loop**, дальше поблочно снимаем «страховочные колёса» → в
пределе 100%-автономный бот. Порядок: раньше — **выше КПД, ниже Сложность, ниже Риск**
(+ зависимости). Содержание блоков уточняется по мере доводки продуктов и возрастных
критериев (см. «Задачи»), но каркас от этого не меняется.

**Блок 0 — Платформа** (фундамент, без него ничего не работает): канал MAX (Wazzup) · роутер
+ дедуп/идемпотентность · чтение/запись Fitbase + карточка + краткосрочная память · зона-бота
gating (лид vs клиент) · механика эскалации (Task + TG-алерт) + ручной режим.

**Функциональные блоки:**

| Блок | Снимает с продавца | КПД | Сложность | Риск |
|---|---|---|---|---|
| F1 Мгновенная реакция + квалификация | ответ ≤5 мин, первичный сбор | выс | низ | низ |
| F2 Консультация / FAQ | типовые вопросы | выс | сред | низ–сред |
| F3 Запись на пробную (+оплата 900₽, дедлайн/авто-снятие) | подбор времени, запись, контроль оплаты | выс | сред | сред |
| F4 Напоминания о пробной | борьба с неприходами | выс | сред | низ–сред |
| F5 Неприход / перезапись | возврат в воронку | сред | сред | низ |
| F6 Дожим после пробной (продажа абонемента) | главная продажа | выс | выс | выс |
| F7 Оформление после покупки (email → карточка/ЛК) | добор данных, граница лид→клиент | сред | низ | низ |
| F8 Реактивация молчащих | прогрев, «напомнить о себе» | выс | выс | выс |
| F9 Анализ звонков (STT, Мегафон) | прослушивание/конспект | сред | выс | низ |

**Порядок (волнами):**
1. **Платформа + F1** (+ минимум F2) — мгновенная реакция и квалификация, остальное на человека. Макс. КПД сейчас, мин. риск, обкатка платформы на живом MAX.
2. **F2 + F3** — ответить и записать на пробную (с оплатой). Основная до-КЭВ работа.
3. **F4 + F5** — напоминания (режут неприходы) и неприход/перезапись.
4. **F6 → F7** — дожим после пробной и оформление купивших (F7 зависит от появления покупателей).
5. **F8** — реактивация молчащих (после схемы касаний и анти-спама).
6. **F9** — анализ звонков; внутреннее, можно **параллельно** в любой момент. Позже — аналитика/дашборд.

Доп. каналы (полноценный WhatsApp, затем Telegram, VK) подключаем по мере надобности —
логика агента от канала не зависит.

## Блок 0 — сводно (на одном экране)

**Задача Блока 0:** надёжно превратить любое входящее в **правильно опознанного лида с
сохранённой историей** — и либо припарковать на человека, либо (с F1) отдать мозгу и вернуть
ответ. Трубы и предохранители; «ум» — F1.

```
ВХОД — две двери:
  A. клиент написал (MAX/…)            B. заявка в CRM (Тильда/VK Ads/руками)
       ▼ push из канала                     ▼ детект: опрос Fitbase / вебхук Тильды
  [КАНАЛ] приём + перевод ↔ наш формат   [сигнал «новый лид»]
       └────────────────────┬────────────────────┘
                            ▼
      [РОУТЕР] дедуп messageId · опознать → ОДИН лид (найти до создания)
                            ▼
      [ВОРОТА] лид → дальше | действующий клиент → off (к админу)
                            ▼
      [РУЧНОЙ РЕЖИМ] человек за рулём? → молчим (#стоп / #старт)
                            ▼
      [ПАМЯТЬ] блокнот этого человека (один на лида, долгоживущий)
                            ▼
      [РЕШЕНИЕ] Блок 0 → эскалация | F1+ → мозг: ответить / эскалировать
                            ▼
      [ОТВЕТ] → через КАНАЛ; обмен пишем в ПАМЯТЬ

ПОД ВСЕМ: ОКНО FITBASE — единая дверь чтения/записи CRM (турникет ≤20 rps)
СБОКУ:    ЭСКАЛАЦИЯ — Task в Fitbase + TG-алерт кнопкой (#старт — возврат бота)
```

**Сквозные правила:** один человек = один лид (найти до создания) · читать карточку перед
ответом (Fitbase = правда, кэш = запаска) · бот сам выходит на контакт (каскад
MAX→WhatsApp→задача) · идемпотентность + турникет (свой баг страшнее трафика) · бот говорит
с F1 (в Блоке 0 клиенту не пишет) · память ходит за человеком, не за каналом.

## Блок 0 — компоненты (детально)

Блок 0 = трубы и предохранители, не мозг. Проверяется «на трубах» (вход опознан → отфильтрован
→ сохранён → при нужде задача продавцу); **общение с клиентом начинается с F1**. Порядок сборки:
окно Fitbase + канал MAX → роутер → ворота → эскалация + ручной режим → память.

**Шов 0→F1.** В Блоке 0 у бота нет заглушек-ответов клиенту (квитанций не шлём) — бот начинает
говорить на F1. Простое «упала заявка → задача продавцу» делает встроенная автоматизация Fitbase,
не наш бот. Записи Блока 0 (лид-подстраховка, заметка, задача) внутренние, клиент их не видит →
логируем, без воротец-ожидания; реальные воротца-одобрения — на F1 (бот пишет клиенту) и F3
(деньги). Кто подтверждает: поведение перед доверием — приёмка тобой по сценариям; рантайм-надзор
в обкатке — продавец.

**Роутер ≠ ЛЛМ.** Роутер — детерминированный код (дедуп, поиск, диспетчеризация): быстро,
предсказуемо, без токенов. ЛЛМ (Claude) — «мозг», его роутер вызывает на шаге «что ответить/
сделать» (F1+). Блок «РОУТЕР + ИИ-АГЕНТ» на схеме = две части: код-диспетчер + ЛЛМ-агент.

**1. Окно Fitbase** — единая дверь для всего чтения/записи в CRM (вся защита и контроль в одном месте).
- Операции в Блоке 0: найти лид · прочитать карточку · заметка · задача. **Создание лида — только
  подстраховка**, если карточки реально нет: опираемся на авто-создание канала/CRM (дверь B уже
  создаёт лид; Wazzup умеет на первое входящее). Правило: сначала найди, создавай только если точно не нашёл.
- **Дедуп слоёный:** CRM мёржит дубли карточек по телефону (`is_duplicate`) — опираемся на готовое;
  наше — `messageId` (не запустить бота дважды) и связка тредов «один человек» в момент ответа.
- Безопасность: токен в env, не в коде · перс. данные не размазывать по логам/лишним копиям
  (152-ФЗ), но скорость не жертвуем (переписка + кэш-запаска остаются) · необратимые записи
  логируем + идемпотентность · **турникет** = темп-ограничитель + откат на 429 (защита от нашего
  же бага, не от живого трафика) · Fitbase недоступен → падаем на кэш-запаску, бот не валится
  (на кэше — **позже, к F3**: ограничить необратимые/денежные действия, т.к. снимок может устареть).

**2. Роутер (вахта на входе)** — диспетчер на каждое входящее: решает путь, сам не отвечает.
- Обе двери (A — сообщение из канала, B — сигнал «лид появился») входят в одну точку.
- **Идемпотентность** по `messageId`: повтор вебхука молча выкинуть (старый баг — ретраи Wazzup
  плодили дубли ответов).
- **Опознать → один лид** (найти до создания).
- **Дирижёр станций:** ворота → ручной режим → память → решение; держит порядок и «СТОП»
  (ворота «действующий клиент» → бот off).
- Безопасность: проверять подлинность вебхука (подпись/секрет — дверь публичная) · идемпотентность
  · не плодить лиды · падать безопасно (ошибка → лог + эскалация).

**3. Канал (транспорт, MAX через Wazzup)** — труба наружу + переводчик формата.
- Слышать (вебхук) / говорить (Wazzup API) / **переводить** Wazzup-формат ↔ наш внутренний (мозг
  про вендора не знает → новый канал = новый переводчик, ядро не меняется). **Метим свои отправки**
  (для ручного режима).
- Дубли: **быстрый ack** (200 сразу, обработка асинхронно) + дедуп `messageId` — иначе ретраи
  Wazzup плодят дубли (старый баг).
- Безопасность: публичная дверь — проверять подпись/секрет (дверь живёт здесь, роутер получает
  уже проверенное) · ключ Wazzup в env · входящее — недоверенный ввод (вкл. prompt-injection,
  лечим на F1) · перс. данные не в логи.

**4. Краткосрочная память** — блокнот диалога, один **на человека** (якорь — телефон/карточка),
долгоживущий (переживает паузы в недели). Живёт у нас (Redis), не в Fitbase.
- Дописывает каждое сообщение (вкл. реплики, пока бот молчит); хранит между сессиями; отдаёт
  контекст (в Блоке 0 — держим наготове, пользуется мозг F1). Опирается на ключ-личность роутера.
- **Хранить всё, но в модель — только нужное** (свежий срез + сводка): весь архив = дорого/медленно
  (старый бот, prompt caching). Краткосрочная (транскрипт) ≠ долгосрочная (портрет/эпизоды — F1+).
  Ключевые факты дублируем в Fitbase `Note`.
- Безопасность: **главный склад перс. данных** (152-ФЗ: срок хранения, сервер в РФ, доступ) · не
  терять историю на сбое (персистентность/бэкап) · память верна настолько, насколько верен ключ
  (рискует расколоть/склеить людей — как у ворот).

**Перед сборкой — что добыть/подтвердить (ворота к коду):**

*Доступы / ключи (добыть):*
- Fitbase: API-токен + `domain` + тестовый домен-песочница.
- Wazzup: аккаунт/ключ + подключённый MAX-канал + тестовый канал.
- Telegram: бот для алертов + chat id продавца. Redis. Anthropic-ключ (уже для F1).

*Fitbase — подтвердить по API:*
- Формат входа (Bearer + `domain`), срок жизни токена.
- Поиск лида по телефону / id канала; реальные поля каналов (`phone, max, vk, …`).
- Как отличить действующего клиента (активный `ClientContract`).
- Фильтр по `updated_at` (радар / опрос двери B).
- Поля задачи (`responsible` = id продавца — узнать; `available_to`, `description`), `Note`.
- Что/как авто-создаёт связка Wazzup→Fitbase (чтоб «создание лида = подстраховка»).
- Что делает сам с дублями (`is_duplicate`). Заодно: есть ли вебхук на оплату (для F3).

*Wazzup / MAX — подтверждено по докам (2026-05-29), осталось проверить на живом payload:*
- Вебхук Wazzup отдаёт: `messageId` (uuid), `chatType` (вкл. `max`), `chatId`, `contact.phone`
  («для Telegram и MAX»), `text`, `contentUri`, `type`. ⚠️ Телефон для MAX — **проверить на реальном
  входящем** (MAX сам отдаёт номер только если юзер «поделился контактом»; вероятно Wazzup подтягивает,
  но не гарантия). Опознавалка — **пул** (`chatId`/MAX `user_id` + имя есть всегда; телефон когда есть).
- Отправка: по `chatId` **или `phone`/`username` для MAX** → дверь B (писать по номеру) работает.
- Подлинность вебхука: заголовок `Authorization: Bearer <crmKey>` (общий секрет, мы задаём; HMAC нет).
- Реплики продавца / метим свои: поле `isEcho` (+ `authorId`/`authorName`) — закрыто.
- **Право писать первым зависит от тарифа Wazzup** (на «Inbox» нельзя; «Старт» — есть, но лимит ~50
  диалогов/мес → мало): нужен тир с API + write-first без узкого лимита (**Pro/Max**), уточнить лимит
  первых сообщений/мес при подключении.
- Тильда → наш вебхук вторым адресом (лендинг твой — настроить).
