## Контекст

Строим автономного AI-агента продаж для детского центра «Мама рядом» (Mama Ryadom), Уфа.
Бот ведёт клиента от заявки до зачисления, **обрамляя КЭВ** (см. ниже): общается в
мессенджерах, помнит контекст между сессиями, ведёт карточку в CRM, ставит задачи,
бронирует/перезаписывает пробные, анализирует записи звонков продавца.

- **Ниша:** B2C, детский центр, дети **1,5–4 года** (черновик заказчика; по факту — по нескольким критериям). Три направления —
  **гимнастика** («Мама и малыш»), **беби балет**, **сенсорика**; детали в разделе «Продукты и возрастные критерии». Цикл сделки длинный и рваный (заявка → дозвон со
  2–3 раза → молчит днями → просит позвонить → ушёл подумать → записался → не пришёл →
  перезапись → уехал на 2 недели → и только потом купил). Это закладываем в память и задачи.
- **Команда:** один продавец. Маршрутизация/распределение заявок не нужны.
- **Масштаб:** пока один бизнес («Мама рядом»). Мультитенант не закладываем.
- **Backend/CRM:** **Fitbase** (`https://api.fitbase.io/api/v2`). Не Firebase — это была
  опечатка. Старая CRM `lk.ufagymnastics.ru` выводится из эксплуатации (см.
  `old-bot-architecture.md` как легаси-референс).

**Подход к автономии:** цель — полная автономия, но строим поэтапно через
**human-in-the-loop**: бот действует/предлагает, продавец может вмешаться. «Страховочные
колёса» снимаем по мере роста доверия. Каркас под автономию ≈ +20% к версии «помощник»
(не «с нуля автономного»). Полную автономию НЕ делаем условием запуска.

## Роль бота — «обрамляет КЭВ»

**КЭВ** (ключевой этап воронки) = **пробная тренировка с презентацией** — момент, где
продаётся основной продукт (абонемент).

**Пробная — средство, не обязательный барьер.** Цель — продать абонемент; пробная лишь
помогает. Готов лид оформить абонемент **сразу, без пробной** — бот продаёт сразу, не гонит
через пробную. Редко, но бот не должен «заставлять» проходить пробное.

- **До КЭВа — целиком бот:** реакция на заявку ≤5 мин, квалификация, «продать» пробную и
  записать. Здесь главная зона автономии.
- **КЭВ — оффлайн, человек:** саму пробную ведёт тренер/продавец. Бот не внутри.
- **После КЭВа — снова бот:** подтверждение прихода, догрев «пришёл → думает → зачислен»,
  обработка неприхода и перезаписи.

## Зона бота — только продажи (лиды), не действующие клиенты

Бот работает **только с лидами / потенциальными клиентами**. С действующими (ходящими)
клиентами он не общается и не выполняет по ним действий — это отдельная база, **зона админа**.

**Роли (для ясности терминологии):** действующие (платящие, ходящие) клиенты — забота
**админа**; заявки и те, кто ещё не оплатил абонемент — забота **продавца**. Бот **занимает
роль продавца** (лиды / неоплатившие).

- На входе в диалог бот **проверяет надёжные идентификаторы** (телефон / id канала: MAX, VK,
  TG) по базе действующих клиентов Fitbase (напр. активный `ClientContract`).
- Совпало по надёжному ключу → действующий клиент: чат уходит **админу**, бот **отключается**
  в этом чате (не тратит ресурсы). **По имени НЕ сверяем** (решение 2026-05-29): тёзок слишком
  много («куча Гузелей»/Елен), ложное отключение лида дороже редкого случая «клиент написал с
  нового номера» — такого бот спокойно ведёт как лида, без тихих заметок админу.
- Когда лид **становится** клиентом (купил абонемент): на подтверждённой оплате — **handoff
  продавцу** (задача+алерт «проверь оплату, дальше веди сам»). Это граница лид→клиент. Продажу
  (дожим до покупки) ведёт бот (F6); пост-продажный онбординг (связать с тренером, «теперь
  вопросы сюда») = **F7**, пока вручную человеком.
- **Бывший клиент вернулся** (абонемент кончился) — снова **лид** (бот прогревает), но
  карточка/память помнят, что уже ходили. Кейсов мало.

## Сквозные инварианты (соблюдает любой блок)

Правила безопасности и поведения, не зависящие от блока. Детали — в `docs/`, отслеживание
внедрения — в [`docs/tasks-open.md`](docs/tasks-open.md).

- **Один человек = один лид** — найти карточку до создания, не плодить дубли. Имя — только слабая подсказка, не ключ автосклейки.
- **Fitbase = источник правды**, кэш — только запаска (необратимое/деньги на кэше не делаем).
- **Бот говорит с F1** — в Блоке 0 клиенту не пишет.
- **Память за человеком, не за каналом.**
- **Зона бота — только лиды** (см. выше): действующий клиент → бот off, к админу; по имени не сверяем.
- **Бот сам выходит на контакт** (каскад MAX→WhatsApp→задача; задача продавцу — последний шаг, не дефолт).
- **Турникет Fitbase** ≤20 rps + бэк-офф на 429; **идемпотентность** по `messageId`.
- 🔴 **Claude только предлагает — код проверяет и исполняет** любые действия с деньгами/скидками/этапами/бронью (LLM не дёргает кнопки напрямую).
- 🔴 **ПД маскируем перед отправкой в LLM:** телефон/фамилия в Claude не уходят; имена для общения — ок. Опознание «кто это» — в нашем коде.
- 🔴 **Стоп-кран всегда:** эхо-фильтр (не реагировать на свои и продавца сообщения) + проверка перед каждой отправкой + автостоп по счётчику. Бот не должен «строчить сам с собой».
- 🔴 **Единый часовой пояс — Уфа (UTC+5)** для всех дедлайнов/напоминаний/тихих часов.

## Архитектура — 3 части

```
┌─────────────────────────────────────────────┐
│              КАНАЛЫ (транспорт)              │
│   MAX 60% · WhatsApp 30% · TG 5% · VK 5%     │
│        Wazzup (агрегатор)  ·  Звонок         │
└───────────────────┬─────────────────────────┘
                    │ входящее (push из канала)
┌───────────────────▼─────────────────────────┐
│         РОУТЕР + ИИ-АГЕНТ  (наш слой)        │
│  · идентификация лида по каналу, дедуп        │
│  · LLM (Claude) + инструменты                 │
│  · память: кратко / долго / эпизод + ретривал │
│  · планировщик проактива, опрос Fitbase       │
│  · решает: ответить сам или эскалировать      │
└──────┬───────────────────────────┬───────────┘
       │ читать / писать            │ задача + алерт
┌──────▼──────────────┐   ┌─────────▼───────────┐
│  FITBASE (CRM,       │   │   ПРОДАВЕЦ (1 чел.) │
│  источник правды)    │   │  Fitbase Task +     │
│  лид, воронка,       │   │  Telegram-алерт     │
│  оплаты, брони,      │   └─────────────────────┘
│  посещения, задачи   │
└──────────────────────┘
```

## Стек

- **Python**; **LLM Claude** (Anthropic API, tool use, prompt caching) — рабочая модель диалога **Sonnet** (Opus не нужен; Haiku для тривиальных под-шагов — на будущее).

### Как добавлять и поддерживать документацию (конвенция проекта)

- **`CLAUDE.md` держим лёгким** — только ядро (оно грузится в каждую сессию). Всё развёрнутое живёт в `docs/`.
- **Новый блок (напр. F1)** → заводим отдельный файл `docs/f1-….md`, складываем туда детали и **добавляем ссылку в список выше**. Так проект растёт, не раздувая `CLAUDE.md`.
- ⚠️ **Ссылки — только обычным markdown** `[текст](путь)`. **Не** писать `@путь`: знак `@` = авто-загрузка файла в контекст при старте сессии (вернёт раздувание до прежних размеров). Обычная ссылка читается **по требованию**, когда реально нужна.
- **Перенос текста между файлами — дословный копипаст** (надёжнее через извлечение строк, не перепечатывание): ничего не теряем; смысловые правки — отдельным осознанным шагом.
