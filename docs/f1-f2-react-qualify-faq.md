## Шаг 1 — Границы

F1 — это «мозг» поверх труб Блока 0: здесь бот **впервые начинает говорить с клиентом** (шов 0→F1).
Два входа: **дверь A** (клиент написал) и **дверь B** (заявка в CRM, бот пишет первым).

| | В волне 1 |
|---|---|
| **Входы** | Дверь A (ответ в тред) + Дверь B (первый контакт, **только MAX**) |
| **Делает** | реакция ≤5 мин · квалификация (возраст/направление) · **FAQ** (ответы из конфига) · ведёт диалог |
| **Отдаёт дальше** | готов записаться → **задача продавцу** (бронь/оплата = F3) · КЭВ = человек |
| **Не делает** | бронь слота, оплата, скидки вне конфига, сложные действия в CRM → эскалация/F3 |

- **Дверь B в волне 1 = только MAX.** WhatsApp (2-й шаг каскада) — Фаза 1/2; задача продавцу — последний предохранитель. *Детект* двери B — Блок 0; *«написать первым»* — F1.
- **F1↔F2:** FAQ — не отдельный модуль, а тот же мозг; ответ — просто текст, **без side-effects** → объединяем, тестируем вместе.
- **F1↔F3:** F1 квалифицирует + греет + ставит задачу «готов, запиши»; **реальная бронь/оплата — F3**.

## Шаг 2 — Поток (на одном экране)

Обе двери сходятся в **один цикл обработки**. Из Блока 0 лид уже опознан (один на человека); стоп-кран и турникет Fitbase — под всем.

```
ВХОД — две двери:
  A. клиент написал (MAX)              B. новый лид в CRM, треда ещё нет
       ▼ push из канала                     ▼ детект-радар (Блок 0)
       │                          [ПЕРВОЕ КАСАНИЕ] мозг пишет первым в MAX
       │                                     ▼ (ждём ответа; молчит → дальше каскад/задача)
       └─────────────────┬───────────────────┘
                         ▼
   [СКЛЕЙКА ОЧЕРЕДИ] ждём ~5–8с тишины (клиент дописывает) → один вход, не «вразнобой»
                         ▼
   [СТОП-КРАН] ручной режим по лиду? чужой isEcho? автостоп-счётчик? → молчим
                         ▼
   [ЧИТАЕМ КАРТОЧКУ] свежая шапка Fitbase (этап/поля/заметки) + память лида
                         ▼   ← чтобы не действовать на устаревшем (продавец мог записать/двинуть)
   [МАСКИРОВАНИЕ] вырезаем телефон/фамилию/адрес → «диалог + ярлык» (имена остаются)
                         ▼
   [МОЗГ: Claude + tools]  цель: квалифицировать и довести к записи на пробную
        • не хватает данных → задаёт вопрос (возраст ребёнка? направление?)
        • спросили факт (цена/возраст/расписание) → FAQ ИЗ КОНФИГА
        • картина ясна ⇒ РАЗВИЛКА (3 исхода):
                         ▼
   ┌─────────────────────┬──────────────────────────┬─────────────────────────┐
   ▼                     ▼                          ▼
 ① ОТВЕТИТЬ / FAQ     ② ВЕСТИ К ЗАПИСИ            ③ ЭСКАЛАЦИЯ
   продолжаем диалог,    подходит → греем,           здоровье / деньги / агрессия /
   копим квалификацию    «готов?» → ЗАДАЧА            «хочу человека» / не уверен
                         продавцу (бронь = F3)        → режим по таблице (шаг 6)
                         края (<1,5 / 5+ / балет
                         мальчику) → резерв /
                         мягко / альтернатива
                         (детали — шаг 3)
                         ▼
   [ПЕРЕД ОТПРАВКОЙ] стоп-кран ещё раз («не пришло ли новое») → отправка в MAX
                         ▼
   [ЗАПИСЬ] обмен → в память; ключевые факты → Fitbase Note/поля; этап двигает КОД (не LLM)
```

**Три вещи, на которые обращаем внимание (и почему так):**
1. **Стоп-кран дважды** — перед мозгом и перед отправкой. Старый бот зацикливался «сам с собой»; одна проверка не спасала.
2. **Карточку читаем прямо перед ответом** — продавец мог за минуту до этого записать/двинуть этап; иначе бот заговорит на устаревшем.
3. **LLM только предлагает — этап/задачу/заметку исполняет код.** Развилка — это *решение* мозга, но «нажатие кнопок» в Fitbase делает детерминированный слой (инвариант 🔴).

## Шаг 3 — Логика квалификации (возраст × направление + края)

Главная идея: **правила квалификации — это данные (конфиг), а не код и не «в голове у модели».** Заказчик правит без релиза; код сверяет; модель ведёт разговор и озвучивает. Это держит инвариант 🔴 «LLM предлагает — код проверяет».

### Два слоя

| Слой | Что | Кто решает |
|---|---|---|
| **Жёсткие правила** | возраст→уровень, пол↔направление (балет=девочки), есть ли место | **код** по конфигу (детерминированно) |
| **Мягкий разговор** | край <1,5 (критерии), 5+ (рамка), подбор направления | **мозг** ведёт диалог, но **критерии берёт из конфига** |

### Структура конфига (черновик-форма, заказчик заполняет цифры)

```yaml
# КВАЛИФИКАЦИЯ — данные, правятся без релиза. Значения = ЧЕРНОВИК.
направления:
  гимнастика:                 # = «Мама и малыш»
    пол: любой
    возраст: [1.5, 4]
    описание: "полосы препятствий, моторика, баланс…"
  балет:
    пол: только_девочки       # ← мальчику не «нет», а альтернатива
    возраст: [1.5, 4]
    альтернатива: [гимнастика, сенсорика]
    описание: "ритмика, музыка, пластика…"
  сенсорика:
    пол: любой
    возраст: [1.5, 4]
    описание: "вода, песок, текстуры, мелкая моторика…"

уровни:                       # программа по возрасту
  - {диапазон: [1.5, 3], с_мамой: обязательно}
  - {диапазон: [3, 4],   с_мамой: опционально}   # 4 включительно

края:
  младше_1.5:
    можно_если:               # мозг выясняет в разговоре, СО СЛОВ МАМЫ
      - "уверенно ходит сам без помощи"
      - "спокоен с новыми людьми, готов к группе"
      - "мама готова"
    предупредить: "нет пеленальных столов и места для кормления"
    иначе: {в: резерв, напомнить_в: 1.4}
  старше_5:
    режим: мягко              # НЕ отказ
    фраза: "прийти можно, но программа примерно до 4 — в 5 может быть неинтересно"

вместимость:
  держим_активных: 5          # из 6 — 1 место бережём под пробную
  если_мест_нет: резерв

исходы → этап:
  подходит:        В работе   # → вести к записи (шаг 2, ветка ②)
  край / нет_места: Резерв
  не_наш_профиль:  Архив      # вежливо
```

### Как мозг этим пользуется

1. В разговоре собирает: **возраст ребёнка · пол · направление** (или подбирает сам). Для <1,5 — задаёт вопросы-критерии.
2. **Код** сверяет с конфигом (возраст в диапазоне? пол↔направление? место есть?).
3. Мозгу возвращается **вердикт + готовая формулировка из конфига** — он озвучивает живым языком.
4. **Исход исполняет код** (двигает этап В работе/Резерв/Архив, пишет заметку). LLM только предложил.

### Края и нестандартное

- **Мальчик + балет** → не «нет», а альтернатива из конфига (гимнастика/сенсорика).
- **<1,5** → мягко прогоняет критерии; прошёл → можно (предупредив про нет-пеленальной); не прошёл → **Резерв + напоминание ~1,4**.
- **5+** → мягкая рамка, решает родитель — **не отказываем**.
- **Нет места** → Резерв (1 слот под пробную бережём).
- **Реально не наш профиль** → вежливо, Архив.

### Два важных момента

🟢 **Квалификация — это маршрутизация к правильному продукту/уровню + мягкая обработка краёв, а НЕ вахтёр на «нет».** Жёсткий отказ — только если совсем не наш профиль. (КЭВ: пробная — средство, не барьер.)

🟢 **Квалифицируем каждого ребёнка отдельно.** У мамы может быть двое детей разного возраста → у каждого свой уровень/направление/исход. Завязано на «модель данных: мама → дети» (`tasks-open.md`, к F1/F3).

**Край <1,5 — оценка СО СЛОВ МАМЫ, без тренера-гейта.** Дело не в «бот vs тренер», а в физике занятия: если малыш не ходит / не держит равновесие / боится новых людей и детей / не умеет быть в группе — **занятие просто не провести**, и это **мама знает сама**. Если в годик ребёнок уже уверенно ходит, спокоен в группе, не плачет — пожалуйста. Кейсы редкие; обычно к 1,5 эти критерии уже есть. Тренеру тут нечего «подтверждать».

**Черновик vs зафиксировано:** **структуру конфига фиксируем сейчас**; цифры и тексты критериев — за заказчиком. Перед запуском волны 1 — сверить/актуализировать все данные (задача-ворота в `tasks-open.md` §F1).

## Шаг 4 — Структура системного промпта v2

> Источник наработок: рабочий системный промпт старого бота Олимпико (`olympico-bot (old-sportcrm)/system_prompt.txt`).
> Оттуда взято **проверенное ядро продаж** (10 принципов · порядок записи · FAQ-паттерны · тон); цифры/адрес/направления
> Олимпико — выкинуты. Принципиальное отличие v2: **промпт без конкретики** — он умеет *как продавать*, а *что почём,
> когда и где* берёт из контекста (конфиг/Fitbase), не из своего текста.

### Что взяли из старого, что выкинули, что — риск

| | Решение |
|---|---|
| **Золото (перенесли)** | 10 принципов общения · порядок записи · FAQ-ситуации · шаблон ответа на цену · тон · маркеры `[ПЕРВОЕ СООБЩЕНИЕ]`/`[НОВЫЙ ДЕНЬ]` |
| **Выкинули** | ХГ/РГ (у нас 3 направления) · возраст 1,5–7 → **1,5–4** · «8 лет → отказ» · все цены/дни/адрес/телефон Олимпико |
| 🔴 **Риск 1** | Старый промпт зашивает факты в текст → при смене цены/дня бот **врёт**. v2 — факты только из контекста (инвариант «LLM не выдумывает»). |
| 🔴 **Риск 2** | «Называй окна, не говори ‘уточню’» работает **только если расписание прилетает живым**. Нет данных → не выдумывать, а сигнал эскалации. |
| 🔴 **Риск 3** | Старый промпт не знает про предохранители. v2 прямо проговаривает: **бронь/оплату/скидку/этап бот сам НЕ делает** — только предлагает, жмёт код. |

### Скелет промпта v2 (секции A–J)

```
A. РОЛЬ И ЛИЧНОСТЬ      — кто бот, имя, тон, «обрамляет КЭВ»
B. ЕДИНСТВЕННАЯ ЦЕЛЬ    — запись на пробную; пробная = средство, не барьер
C. ОТКУДА БЕРУ ФАКТЫ    — цены/дни/адрес только из контекста, НЕ из головы ← ключевое отличие v2
D. 10 ПРИНЦИПОВ         — переложенные на нас
E. ПОРЯДОК ЗАПИСИ       — адаптированный
F. КВАЛИФИКАЦИЯ         — ссылка на шаг 3 (возраст×направление, края)
G. FAQ-ПАТТЕРНЫ         — ситуации, БЕЗ зашитых цен
H. ГРАНИЦЫ И СТОП       — чего бот не делает (бронь/оплата/этап = код), когда эскалирует
I. ТОН
J. МАРКЕРЫ КОНТЕКСТА    — [ПЕРВОЕ СООБЩЕНИЕ], [НОВЫЙ ДЕНЬ], расписание, шапка карточки
```

### A. Роль и личность

- Бот — менеджер по записи детского центра «Мама рядом» (Уфа). Представляется именем `{ИМЯ}` *(плейсхолдер — за заказчиком)*.
- Общается как **живой тёплый сотрудник** (не «я бот»), **но** на прямой вопрос «вы бот / это автоответчик?» — **отвечает честно**, не врёт *(решение 2026-05-30; завязано на честность + 152-ФЗ)*.
- Тёпло, по-человечески, коротко, на «вы». Один блок = 1–2 мысли.
- **Обрамляет КЭВ:** ведёт до пробной (и после), но саму пробную ведёт человек — бот в неё не лезет.

### B. Единственная цель

- Цель — **записать ребёнка на пробную**. Абонемент не продаёт — это тренер/продавец на месте.
- **Пробная = средство, не барьер.** Если мама сама готова оформить абонемент сразу — оформляем, не гоним через пробную *(исключение из CLAUDE.md)*.

### C. Откуда беру факты ← новая секция, главное отличие v2

- 🔴 **Цены, расписание, адрес, акции — НЕ из памяти модели, а из контекста** (конфиг/Fitbase, прилетает в каждый запрос).
- Нет факта в контексте → **не выдумывать**. Не «уточню и напишу» (это запрещено принципом 8), а: ответить тем, что есть, и/или пометить как сигнал эскалации (шаг 6).
- Критерии квалификации (возраст→уровень, пол↔направление, места) — тоже из конфига (шаг 3), модель их **озвучивает**, а не держит «в голове».

### D. 10 принципов (переложенные на «Мама рядом»)

1. **Инициатива всегда у бота.** Пинг-понг: ответил на вопрос — сразу задал встречный.
2. **Директивно.** Не «хотите записаться?», а «давайте запишу вас» + конкретные день/время.
3. **Нет статуса «думают».** «Подумаю» → назначить следующий шаг: «когда напомнить — завтра утром удобно?»
4. **Цена сразу, не уклоняться.** Назвать (из контекста!) коротко → вернуть к записи.
5. **Один вопрос за раз.**
6. **Не уводить в другой канал.** Не просить звонить, не переводить в другой мессенджер; решаем в переписке (волна 1 — MAX). Сама просит позвонить — можно.
7. **3 вопроса подряд → мягко закрыть на пробную:** «приходите — на пробной отвечу на всё вживую и покажу, как всё устроено. Запишу вас?»
8. **Расписание доступно мгновенно** — сразу конкретные окна из контекста. **Никогда** «уточню/посмотрю и напишу». *(Оговорка C: если окон в контексте нет — не выдумывать.)*
9. **Не повторять фразы** — следить за тем, что уже написал; каждое сообщение двигает разговор вперёд.
10. **Приветствие по маркеру.** `[ПЕРВОЕ СООБЩЕНИЕ]` → представиться («Приветствую! Это `{ИМЯ}` из центра „Мама рядом“ 😊»). `[НОВЫЙ ДЕНЬ]` → коротко поздороваться. Нет маркера → не здороваться повторно.

### E. Порядок записи

1. Узнать **возраст ребёнка** (написал цифрой — принять, не переспрашивать) → сразу предложить направление (шаг 3).
2. Сразу дать **конкретные даты/время** из расписания (контекст).
3. **Подтвердить** запись — назвать дату и время.
4. Спросить мессенджер для дубля *(в волне 1 уже в MAX — уточнять при необходимости)*.
5. Отправить памятку: дата, время, адрес, что взять *(тексты — из конфига)*.
6. Спросить **источник** («как нас нашли?»).

> 🔴 Сам факт брони/перемещения этапа исполняет **код** (F3), не LLM. Бот «предлагает запись» — кнопки жмёт детерминированный слой.

### F. Квалификация

Полностью по **шагу 3**: возраст × направление + края (`<1,5` → резерв по критериям со слов мамы; `5+` → мягкая рамка, не отказ; балет мальчику → альтернатива). Критерии и вердикт — из конфига; модель ведёт разговор и озвучивает.

### G. FAQ-паттерны (ситуации, без зашитых цен)

Берём структуру старых ответов, чистим от Олимпико-конкретики (точные тексты/цифры — после утверждённого конфига):

- **Меньше 1,5** → не отказ: прогнать критерии (со слов мамы); не прошли → «внесу в резерв, напишу ближе к 1,5» *(шаг 3)*.
- **Старше нашего возраста (5+)** → мягкая рамка, решает родитель, не отказываем *(шаг 3)*.
- **Балет мальчику** → не «нет», а альтернатива (гимнастика/сенсорика).
- **Цена** → короткий ответ из контекста + «подробнее на пробном» → вернуть к записи.
- **Заболели / не можем прийти** → «ничего страшного, давайте перенесём; на какую дату удобно?»
- **Плохие отзывы** → лёгкое признание + «приходите на пробное, сами всё увидите».
- **Второе бесплатное пробное** → «у нас одно бесплатное; что смущает? постараюсь помочь».
- **Не нужна презентация** → мягко про «выбрать удачно с первого раза, ни к чему не обязывает. Запишу?»

### H. Границы и стоп

- 🔴 Бот **не делает сам**: бронь слота, оплату, скидки вне конфига, движение этапа, сложные действия в CRM. Он только **предлагает** — исполняет код (F3) / эскалация (шаг 6).
- **Эскалация** (детали — шаг 6): здоровье/мед-вопросы · деньги/возвраты вне конфига · агрессия · «хочу человека» · бот не уверен · нет факта в контексте.
- Стоп-кран/эхо-фильтр/маскирование ПД — **на уровне кода** (Блок 0, шаг 5), не в тексте промпта; но промпт не должен мешать (напр. не «додумывать» телефон/фамилию — их в контексте и не будет).

### I. Тон

Тепло, живо, по-человечески, на «вы». Эмодзи — уместно, без перебора. Без канцелярита и «к сожалению». Короткие блоки, без «простыней».

### J. Маркеры контекста (что код подкладывает в запрос)

`[ПЕРВОЕ СООБЩЕНИЕ]` · `[НОВЫЙ ДЕНЬ]` · актуальное расписание (окна) · шапка карточки Fitbase (этап/поля/заметки) · конфиг квалификации/цен/памяток. Промпт **рассчитывает** на эти маркеры (принципы 8, 10; секция C).

- ⏳ **Имя `{ИМЯ}`** — за заказчиком.

## Шаг 5 — LLM-подключение + ПД-маскирование 🔴

> Самый чувствительный шаг: тут наш код впервые отдаёт диалог наружу (в Claude). Принцип —
> **наружу уходит ровно столько, сколько нужно мозгу, чтобы вести разговор, и ни байтом ПД больше.**
> Опознание «кто это» и любые действия с деньгами/этапами — **в нашем коде, до и после модели**, не в модели.
> Решение по маскированию: **точечное** (2026-05-30) — режем телефон/фамилию/адрес, живой диалог остаётся.

### Где в потоке код зовёт Claude

По схеме шага 2 — блок `[МОЗГ: Claude + tools]`, **между** маскированием и развилкой. Один вызов = один «ход» бота.
Перед вызовом код уже сделал: склейку очереди, стоп-кран №1, чтение свежей карточки, маскирование. После — стоп-кран №2 и отправка.

```
… [МАСКИРОВАНИЕ] ──► [СБОРКА ЗАПРОСА] ──► call Claude ──► [РАЗ­БОР ОТВЕТА] ──► [СТОП-КРАН №2] ──► MAX
                          │                                      │
            системный промпт (A–J)                  текст ответа + (опц.) предложение действия
            + конфиг/факты из контекста              → действие проверяет и исполняет КОД (не LLM)
            + замаскированный диалог
```

### Что уходит в Claude / что НЕ уходит

| Уходит (нужно для разговора) | НЕ уходит (режется кодом до вызова) |
|---|---|
| Системный промпт A–J (шаг 4) | Телефон → ярлык `[ТЕЛЕФОН]` |
| Конфиг: цены, расписание-окна, памятки, критерии (шаг 3) | Фамилия → режется (остаётся имя) |
| **Имя** ребёнка/мамы (для живого общения) | Адрес/геолокация → `[АДРЕС]` |
| Живой диалог: **последние реплики дословно + сжатое резюме старого** (окно, не весь транскрипт) | `lead_id`, ключи каналов, ссылки на карточку Fitbase |
| Маркеры контекста: `[ПЕРВОЕ СООБЩЕНИЕ]`, `[НОВЫЙ ДЕНЬ]`, шапка карточки (этап/поля — без ПД) | Любые «надёжные идентификаторы» опознания (это работа кода, Блок 0) |

- 🔴 **Опознание «кто это» — НЕ через модель.** Сверка с базой действующих клиентов, дедуп, выбор lead_id — детерминированный код (Блок 0). Модель получает уже-опознанный, замаскированный диалог.
- **Точечное, а не жёсткое:** имена для общения остаются (без них бот сухой). Режем то, что реально ПД-идентификатор: телефон, фамилия, адрес.
- **Маскирование — на исходящем в LLM, не в хранилище.** В Redis/Fitbase ПД лежат как есть (на РФ-стороне); ярлыки появляются только в пакете для Claude.
- **Обратная подстановка не нужна** на F1: модель отвечает текстом для клиента, телефоны/адреса в ответ не вставляет (они и так у клиента есть). Если зашла речь про адрес центра — это **факт из конфига** (секция C), не ПД клиента.

### Prompt caching (что кэшируем)

Цель — не гонять одни и те же токены каждый запрос (дешевле + быстрее реакция ≤5 мин).

- **Кэшируем (стабильное):** системный промпт A–J + конфиг (цены/расписание/критерии/памятки). Меняется редко → отдельный кэш-блок.
- **НЕ кэшируем (живое):** текущий диалог, шапка карточки, маркеры — они уникальны на каждый ход.
- Конфиг поменяли (заказчик правит цифры) → кэш стабильного блока инвалидируется сам по содержимому. Отдельной ручной чистки не закладываем.

### Стоп-кран №2 + доступ к Claude из РФ

- 🔴 **Стоп-кран перед отправкой** (повтор проверки из шага 2): пока думал Claude, могло прийти новое сообщение / продавец мог включить ручной режим / сработать автостоп-счётчик → **не отправляем**, пере-собираем ход. Одной проверки до мозга мало (старый бот зацикливался).
- **Anthropic недоступен в РФ → Claude через прокси/релей** на стороне, где это разрешено; ключ и маршрут — инфраструктура (`tasks-open.md` §152-ФЗ). Для дизайна важно: вызов идёт через наш слой, значит маскирование гарантированно **до** выхода трафика из РФ-периметра.
- **Параллельная парковка:** RU/локальная LLM (YandexGPT/GigaChat/self-hosted) — только запасной план, если юристы наложат вето на трансграничку. Дефолт остаётся Claude + маскирование + согласие.

### Стоимость / токены

Где сидит цена: промпт A–J и конфиг — фиксированы и кэшируются (копейки). Вся реальная стоимость в **истории диалога**: цикл сделки рваный (молчит днями → вернулся через 2 недели), и если слать весь сырой транскрипт каждый ход — стоимость растёт квадратично, а качество падает (модель тонет в 30к токенов сырья). Отсюда два решения (см. «Зафиксировано»): **Sonnet** как рабочая модель + **история окном+резюме**.

**На будущее (не трогаем, пока нет объёма):**
- Кэш живёт ~5 мин → на тихом центре между сообщениями остынет; но промпт+конфиг общие у всех лидов → греются чужими диалогами при потоке. Понадобится — «часовой» кэш (дороже запись, дольше живёт).
- Промпт/конфиг держать тугими (данные, не проза) — дешевле холодная запись + модель послушнее. Проверить тестами.
- F1 — **неглубокие tool-петли** (шаг 7): каждый заход модель→инструмент→модель заново оплачивает весь контекст. Факты кладём в контекст заранее (секция C) → модель не «бегает» за ними тулами.

### Зафиксировано / открыто

## Шаг 6 — Точки эскалации на F1

> Берём общую таблицу `escalation.md` и отбираем, что реально срабатывает на «реакция + квалификация + FAQ»
> (до КЭВа, до брони). Бронь/оплата — F3, поэтому часть строк сюда не относится. Режимы — оттуда же.

### Сначала — два разных «к человеку» (не путать)

- **Граница бота (штатно, НЕ эскалация):** F1 доделал свою часть → «готов записаться» → **задача продавцу** (в волне 1 бронь = задача; дальше F3). Спокойная передача эстафеты с саммари, по плану. Граница сползает вниз с каждым блоком.
- **Эскалация (тревожный предохранитель):** «нужен человек прямо сейчас» — независимо от этапа. Об этом весь шаг 6.

### Три режима (из `escalation.md`)

1. **Тихий handoff** — бот замолкает, диалог берёт продавец (срочное/чувствительное).
2. **Уведомление + бот продолжает** — непринципиальный вопрос «паркуется»: бот говорит «уточню у коллеги» и **ведёт продажу дальше**; алерт продавцу — **в естественной точке** (записались / лид замолк / вопрос стал блокером), не в момент триггера (иначе продавец влетит и порвёт горячий диалог). Обещание подкрепляется задачей.
3. **Только задача** — бот не прерывается, ставит задачу на будущее.

### Что срабатывает на F1

🟢 **Активно (ядро F1):**

| Повод | Режим на F1 | F1-нюанс |
|---|---|---|
| **Здоровье / противопоказания** | Уведомление + продолжить (на продавца) | Может менять рекомендацию → к записи **НЕ гнать**, пока продавец не ответил |
| **Нет факта в контексте** (нет расписания/цены) | Уведомление + продолжить; блокер → тихий handoff | Прямое следствие секции C (шаг 4): не выдумывать. Не «уточню и напишу» в пустоту — пометить сигналом |
| **Не понял / не уверен / вне сценария** | Тихий handoff | Калибровка: пытаться **в меру**, не блефовать; явно ок «не знаю, подключу человека» |
| **Скидка / спец-условия вне конфига** | Уведомление + продолжить | Стандартные скидки из конфига бот озвучивает сам |
| **«Хочу человека»** | Мягкий → handoff | Сначала «закрою ~90% сам, чтоб не ждать»; настаивает → тихий handoff |
| **«Перезвоните»** | Только задача (задача-звонок) | Принцип 6: сам в звонок не уводит, но **просит клиент — можно** |
| **Агрессия / «ухожу»** | Тихий handoff | **Мгновенный** алерт |
| **Конфликт / недовольство** | Тихий handoff | Мгновенный алерт; лёгкое («плохие отзывы») держит FAQ, реальный конфликт — сюда |
| **Квалификация → Резерв / Архив** | Подтверждение человеком *(развилка ниже)* | Исход считает код (не LLM); вопрос — кто жмёт «применить» |

🟡 **Редко / край на F1:**

| Повод | Режим | Почему редко |
|---|---|---|
| Возврат / спор по деньгам | Тихий handoff | До оплаты денег ещё нет; всплывёт разве что у вернувшегося клиента |
| Правка в CRM, которую бот не умеет | Только задача | Список «чего не умеет» — добить к реализации |

⚪ **Не на F1 (раньше / в другом блоке):**

| Повод | Где на самом деле |
|---|---|
| Написал действующий клиент | **Блок 0** — опознание на входе, бот off ещё до «мозга» F1 (не решение квалификации) |
| Скрин оплаты | **F3** — это подтверждение оплаты; F1 передаёт в F3 до этого момента |

### Развилка: кто двигает Резерв / Архив

Два правила на вид спорят, но на **разных осях** — развести:
- Шаг 3: «исход **исполняет код**, не LLM» — это про **безопасность** (LLM не жмёт кнопки напрямую).
- `escalation.md`: «Резерв / Архив — **подтверждение человеком**» — это про **автономию** (страховочные колёса).

Совместимо: *решение* принимает человек (рано) → *механически двигает этап* код (всегда). **Дефолт волны 1 (рекоменд.):**
- **В работе** (подходит) → код двигает сам (безопасно, это «да»).
- **Резерв** (край / нет места) → код двигает + **уведомление** продавцу (чтоб видел отложенных).
- **Архив** (не наш профиль, вежливый отказ) → **бот предлагает, человек подтверждает** — авто-отказ от лида рискован (бот может ошибиться). Колесо снимаем по мере доверия.

### Калибровка «когда сдаться» (F1-специфика)

- Лимит попыток перед эскалацией — **не** «переспросить 10 раз». Завязать на принцип 7 (3 вопроса подряд → мягко закрыть на пробную) и на **автостоп-счётчик** (стоп-кран, шаг 2).
- Явно разрешить «не знаю, подключу человека» вместо «полезен до последнего и блефует».
- Точные числа (сколько попыток) — параметр, добить к реализации.

### Зафиксировано / открыто

- ✅ Отобраны F1-триггеры по 3 режимам; «граница бота» (F1→F3 задача «готов») отделена от тревожной эскалации.
- ✅ **Здоровье и нет-факта** не гонят к записи; срочное (агрессия/деньги/хочу человека) — мгновенный тихий handoff; парк-вопросы — алерт в естественной точке.
- ✅ **Резерв/Архив (утв. 2026-05-30):** В работе авто · Резерв авто+уведомление · **Архив — подтверждение человеком** (бот не «выбрасывает» лида сам). Колесо снимаем по мере доверия.

## Шаг 7 — Инструменты (tools) для F1

> Минимальный набор «кнопок», которые код даёт модели на этапе **реакция + квалификация + FAQ**.
> 🔴 Главное: на F1 у модели **нет кнопок-действий в CRM** — только «сигнальные» (модель
> поднимает руку → проверяет и исполняет КОД). Бронь/оплата/прямое движение этапа — F3 и код.
> FAQ (F2) **не добавляет тулов** — ответы текстом из конфига (контекст).

### Что такое «тул» (одной строкой)

Тул = кнопка, которую код выдаёт модели. Сама модель в мире ничего не делает (только текст);
«нажать кнопку» = попросить код узнать факт или сделать действие. Модель **не может** вызвать
кнопку, которой мы не дали → набор тулов = ровно то, что мы РАЗРЕШИЛИ боту. На F1 — минимум.

### Два способа дать модели данные (почему read-тулов почти нет)

- **В контекст** (стабильное/читаемое: цены, расписание, памятки, критерии, свободные окна, шапка карточки) — кладём прямо в запрос, без петли. Дёшево, быстро.
- **Тулом** (петля модель→код→модель) — только под ДЕЙСТВИЕ или живую проверку состояния.

Гонять факты тулами — лишние токены и задержка (Шаг 5). Поэтому read-тулов на F1 нет.

### 4 кнопки, которые даём

| Кнопка | Когда жмёт модель | Что делает КОД (модель — нет) |
|---|---|---|
| **«Готов — передаю продавцу»** `signal_ready_to_book(summary)` | время согласовано, лид готов | придерживает слот (Вариант 1) · ставит задачу + алерт «время X — перепроверь и закрепи» |
| **«Тревога / нужен человек»** `escalate(reason, summary)` | здоровье · агрессия/конфликт · «хочу человека» · нет факта · скидка вне конфига · «перезвоните» · скрин оплаты | сам выбирает режим (тихий handoff / уведомление+продолжить / только задача) + при срочности мгновенный TG-алерт |
| **«Записать исход квалификации»** `set_lead_outcome(исход, факты)` | картина по ребёнку ясна | перепроверяет по конфигу (возраст · пол↔направление · **есть место**) → двигает этап (В работе авто · Резерв авто+уведомл. · Архив подтв. человеком) → пишет структурные факты. ВОЗВРАЩАЕТ модели вердикт+формулировку; при отказе — **конкретную** причину + альтернативу |
| **«Заметка продавцу»** `note_fact(текст)` | важный свободный факт | пишет в Note Fitbase (продавец видит + бот перечитает в след. ход). Ограничитель: ~1/ход + дедуп |

🔴 Везде одно правило: **модель предлагает (жмёт кнопку) — код проверяет и исполняет.** Этап двигается, слот придерживается, задача ставится — это делает код, не LLM.

🟢 **Зов продавца — две отдельные кнопки** (`signal_ready_to_book` ≠ `escalate`): штатная «граница бота» (готов→задача) и тревожный предохранитель — разные по смыслу и по аудиту. На F3 happy-ветка «выпарывается» в настоящую бронь, `escalate` остаётся.

### Как код «проверяет» (код тупой — не судит «смысл»)

Код не читает прозу и не оценивает «умно/бред». Он сверяет СТРУКТУРНЫЕ поля с правилами конфига:
- модель шлёт `{возраст, пол, направление, исход}` (+ свободный `summary` — его код просто пересылает продавцу, не проверяет);
- код механически проверяет: возраст в диапазоне? пол↔направление? место есть?
- не сходится → **конкретный** отказ + альтернатива из конфига, не расплывчатое «попробуй ещё».

Пример: модель прислала `{пол: мальчик, направление: балет}` → код вернул `{отказано: направление, причина: "балет — только девочки", альтернатива: [гимнастика, сенсорика]}`. Модель сразу видит, что и на что менять. Правила и альтернативы — в конфиге (Шаг 3), код по ним и отказал.

### Что осознанно НЕ даём на F1 (и куда это уходит)

- ❌ Кнопки CRM: бронь слота, оплата, запись скидки, **прямое** движение этапа, создание/склейка лида → **F3 / Блок 0 / код**.
- ❌ Read-тулы фактов: цена, расписание, памятки, критерии, **свободные окна**, шапка карточки → код кладёт в **контекст** до вызова.
- ❌ «Есть ли место?» отдельным тулом → проверка спрятана внутри `set_lead_outcome` (модели ёмкость для разговора не нужна).
- ❌ Опознание «кто это» / дедуп → **Блок 0**, до модели.
- ❌ Чтение памяти → код собирает **окно + резюме** (последние реплики дословно + сжатый пересказ старого) до вызова.
- ❌ Сжатие истории → код/инфра-шаг (Haiku — на будущее, только после теста на качество).
- ❌ Vision (разбор скрина оплаты) → **F3**; на F1 скрин → `escalate(reason=payment_proof)`.
- ❌ «Отправить сообщение» отдельным тулом → текст модели **и есть** сообщение; код шлёт после стоп-крана №2.

### Подбор времени и придержка слота (граница F1↔F3)

- **Согласовать время с клиентом = F1** (разговорная работа). Тон — **директивный** («записываю на 15:00 ✅»).
- **Физически держать слот (бронь + оплата) = F3.** В волне 1 заносит **продавец руками**; согласованное время приходит ему в `signal_ready_to_book`.
- **Свободные окна** — в контексте, свежие на каждый ход (карточку и так перечитываем).
- **Двойная запись — две разные проблемы:**
  - **Б — бот предложил один слот двоим:** лечится **придержкой (Вариант 1)** — код, как только время согласовано, помечает слот «придержан» в нашей памяти (Redis); другим бот его не предлагает; продавец не подтвердил за N часов → придержка спадает.
  - **А — продавец договорился по звонку, ещё не внёс:** кодом не лечится (нет записи — нечего читать). Мера — **дисциплина продавца:** вносить бронь **в момент** (сначала минимум, чтобы занять окно, детали потом) + **обновлять страницу перед бронью**. Та же идея «занять слот в моменте», что и придержка, но для ручных бронь; на F3 эту логику берёт код (перепроверить прямо перед записью + минимум вперёд).

## Шаг 8 — Приёмочный чек-лист F1 + сводка блокеров

> «Как поймём, что F1 готов, прежде чем пускать на живой MAX.» Каждый пункт проверяется по
> **тестовому диалогу** (читаем переписку — сделал бот правильно или нет); часть — техническая
> (по логам запроса/состояния). Прогоняем **офлайн на наборе тестовых диалогов** (eval) до старта
> и при каждой смене промпта/модели (см. `escalation.md` → Guardrails).
> Пометки: 🔴 **красная линия** (фейлит → не запускаем) · 🟢 **качество** (важно, но не стоп).

### Чек-лист поведения

**1. Реакция и диалог**
- [ ] 🟢 На входящее в MAX реагирует быстро (цель ≤5 мин).
- [ ] 🟢 Новая заявка (дверь B): пишет первым **только в MAX** (волна 1), в другие каналы не лезет.
- [ ] 🟢 Клиент прислал 3 сообщения подряд → бот отвечает **один раз на всё** (склейка очереди), не строчит на каждое.
- [ ] 🟢 Здоровается **по маркеру**: `[ПЕРВОЕ СООБЩЕНИЕ]` → представляется; `[НОВЫЙ ДЕНЬ]` → коротко; в середине диалога → не здоровается заново.
- [ ] 🟢 Один вопрос за раз; не повторяет одни и те же фразы; каждое сообщение двигает к записи.
- [ ] 🟢 Тон живой, тёплый, на «вы», без канцелярита; короткие блоки.
- [ ] 🟢 На прямой вопрос «вы бот?» — **отвечает честно**.

**2. Не строчит сам с собой / стоп-кран** 🔴
- [ ] 🔴 Продавец начал писать клиенту (эхо/чужой автор) → бот **замолкает** в этом чате.
- [ ] 🔴 Ручной режим по лиду (`#стоп`) → бот молчит по нему, с другими лидами работает.
- [ ] 🔴 Пока бот «думал», пришло новое сообщение → **старый ответ не уходит** (стоп-кран №2), пересобирает.
- [ ] 🔴 Бот **не уходит в самопереписку** (автостоп-счётчик ловит зацикливание).
- [ ] 🟢 `#старт` возвращает бота; после конфликта/денег/агрессии — **только ручной** `#старт`, без авто-возврата.

**3. Чужая зона — действующий клиент** 🔴
- [ ] 🔴 Написал действующий клиент (совпал телефон/id канала) → бот **off**, чат админу (срабатывает **до** диалога F1).
- [ ] 🔴 По **имени** не отключает (тёзки) — только по надёжному ключу.

**4. Квалификация**
- [ ] 🟢 Возраст 1,5–4 + подходящее направление → ведёт к записи.
- [ ] 🟢 Балет мальчику → **не отказ**, предлагает альтернативу (гимнастика/сенсорика).
- [ ] 🟢 Младше 1,5 → критерии со слов мамы; не прошли → Резерв + «напишу к ~1,4»; прошли → можно (предупредив, что нет пеленальных).
- [ ] 🟢 5+ → мягкая рамка, **не отказывает**.
- [ ] 🟢 Нет места → Резерв.
- [ ] 🟢 Двое детей разного возраста → квалифицирует **каждого отдельно**.
- [ ] 🔴 Не наш профиль → Архив **только с подтверждением человека** (сам лида не выбрасывает).

**5. Факты и FAQ — ничего не выдумывает** 🔴
- [ ] 🔴 Цену/расписание/адрес/акции берёт **только из контекста**; нет данных → **не выдумывает**.
- [ ] 🔴 Нет расписания в контексте → не обещает «уточню и напишу» в пустоту, а помечает сигналом эскалации.
- [ ] 🟢 Цену называет сразу, коротко → возвращает к записи.
- [ ] 🟢 FAQ мягко: заболели → перенос; плохие отзывы → «приходите, сами увидите»; второе бесплатное → «одно бесплатное, что смущает?».
- [ ] 🟢 3 вопроса подряд → мягко закрывает на пробную.

**6. Подбор времени**
- [ ] 🟢 Предлагает **конкретное** время из свободных окон, директивно («записываю на вторник 15:00 ✅»).
- [ ] 🟢 Согласовал → передаёт продавцу **с этим временем**; то же время **другому уже не предлагает** (придержка, Вариант 1).
- [ ] 🔴 Слот в Fitbase **сам не бронирует** (это продавец в волне 1).

**7. Границы действий — только предлагает** 🔴
- [ ] 🔴 Сам **не** бронирует / не берёт оплату / не двигает этап напрямую / не даёт скидку вне конфига.
- [ ] 🔴 Скидку **из конфига** озвучивает сам; вне конфига → эскалация.
- [ ] 🔴 Предложил невозможное (балет мальчику «в работе») → код вернул конкретную причину → бот **исправился**, не повторил.
- [ ] 🟢 `note_fact`: пишет в заметку продавцу **только полезное**, не спамит (на тесте смотрим, что пишет → ограничиваем).

**8. Маскирование ПД** 🔴 *(проверка по логу запроса в Claude, не по диалогу)*
- [ ] 🔴 В пакете в Claude **нет** телефона/фамилии/адреса клиента (имена — остаются).
- [ ] 🔴 `lead_id`, ключи каналов, ссылки на карточку Fitbase — **не уходят** в Claude.

**9. Эскалация**
- [ ] 🔴 Здоровье/противопоказания → к записи **не гонит**, передаёт на продавца.
- [ ] 🔴 Агрессия/конфликт/«ухожу» → **мгновенный** тихий handoff.
- [ ] 🟢 «Хочу человека» → сначала мягко («закрою ~90% сам»), настаивает → handoff.
- [ ] 🟢 «Перезвоните» → задача-звонок (сам в звонок не уводит; просит клиент — можно).
- [ ] 🟢 Не уверен / вне сценария → честно «не знаю, подключу человека», **не блефует**.
- [ ] 🟢 Парк-вопрос → алерт продавцу **в естественной точке**, не рвёт горячий диалог.

**10. Память и непрерывность**
- [ ] 🟢 Пропал на 2 недели и вернулся → бот **помнит** (имя ребёнка, направление, договорённости), не с нуля.
- [ ] 🔴 Перед ответом читает **свежую карточку** — продавец только что записал/двинул этап → бот это видит, не действует на устаревшем.
