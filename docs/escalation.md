## Эскалация / handoff (человек ↔ бот)

**Два смысла «передачи человеку» — не путать.** (1) **Граница бота** — штатная передача:
докуда бот доделан, там спокойно отдаёт эстафету продавцу (с саммари); граница **сползает
вниз по воронке** с каждым новым F-блоком. (2) **Эскалация** (ниже) — тревожный предохранитель:
«нужен человек прямо сейчас» на любом этапе, не двигается.

**Эскалация** = бот не решает сам и привлекает продавца. **Три режима** (для разных
триггеров — разные; точную карту «триггер → режим» делаем отдельной итерацией):
1. **Тихий handoff** — бот замолкает, диалог берёт продавец.
2. **Уведомление + бот продолжает** — спорный вопрос «паркуется» на человека: бот говорит
   клиенту «уточню у коллеги, вернусь» и продолжает вести диалог по остальному. **Алерт
   продавцу — в естественной точке, не в момент триггера** (иначе продавец влетит и порвёт
   горячий диалог): бот паркует непринципиальный вопрос, ведёт продажу дальше (подобрать время,
   записать), а задачу+алерт ставит, когда диалог дозрел (записались/оплатили) ИЛИ вопрос стал
   блокером ИЛИ лид замолк — по таймеру, чтобы не потерялось. Если непонятка может изменить саму
   рекомендацию (напр. здоровье) — к оплате не гнать. Срочное (агрессия, деньги, скрин, «хочу
   человека») — мгновенно, это уже тихий handoff. Обещание **обязательно подкрепляется** задачей.
3. **Только задача** — бот не прерывается, ставит задачу на будущее.

**Механика:** `Task` в Fitbase (`responsible`=продавец, `available_to`=срок,
`description`=контекст) + при срочности — **мгновенный алерт в Telegram**: кнопкой с опознавалкой
(`#4232 Маша · оплата`) + контекст (кто/почему/последняя реплика); тап → действия (`#старт` и пр.)
с подтверждением да/нет от случайного нажатия; несколько открытых — список, гасятся по одной.
Один продавец → без маршрутизации. Тот же механизм = **клапан автономии** (по мере доверия триггеры снимаем).

| Повод | Режим | Примечание |
|---|---|---|
| Скидка / спец-условия вне списка | Уведомление + продолжить | Стандартные скидки из списка бот даёт сам (см. «Пробное и оплата») |
| Здоровье / противопоказания | Уведомление + продолжить | На **продавца** |
| Возврат / спор по деньгам | Тихий handoff | Сначала «спасаем» (см. ниже), потом зовём; бот: «я уже позвал человека» |
| Конфликт / недовольство (в т.ч. после пробной) | Тихий handoff | Мгновенный алерт |
| Агрессия / «ухожу» | Тихий handoff | Мгновенный алерт |
| «Хочу человека» | Мягкий | Бот сначала предлагает помочь сам («закрою ~90% вопросов, чтобы не ждать»), при настойчивости — handoff |
| «Перезвоните» | Только задача | Задача-звонок + алерт |
| Не понял / не уверен / вопрос вне сценария | Тихий handoff | См. «Калибровку»: пытаться, но **в меру**, не блефовать |
| Скрин оплаты | Тихий handoff на проверку | Продавец проверяет и **разрешает продолжить** (пока оплату подтверждает человек) |
| Написал действующий клиент | Тихий handoff + бот off | См. «Зона бота» |
| Правка в CRM, которую бот не умеет | Только задача | Список «чего бот не умеет» — добить |
| Резерв / Архив | Подтверждение человеком | Бот предлагает — человек подтверждает |

**Возврат пробной — сначала спасаем, потом возвращаем.** Перед handoff бот уточняет причину:
«совсем неактуально — или просто сейчас неудобно?». Если ребёнок заболел / не успевают на
этой неделе — предлагает **перенос записи**, а не возврат (возврат по закону = заявление +
издержки на переводы; перенос удобнее всем). Возврат — крайняя мера.

**Неприходы — без жёсткого лимита.** Бот перезаписывает сколько угодно, неоплаченная бронь
авто-снимается (`auto_delete_time`). Даже «оплатил → вернули → снова хочет» — пожалуйста.

**Калибровка «когда сдаться»** *(добить)*: задать лимит попыток, прежде чем эскалировать (не
«переспросить 10 раз»), и явно разрешить боту признавать «не знаю, подключу человека» вместо
того чтобы быть полезным до последнего и блефовать.

**Ручной режим:** когда подключился человек — бот замолкает в диалоге **по этому лиду** (не весь
бот: с остальными лидами работает; состояние «ручной/авто» хранится по каждому лиду отдельно).
**Автоглушение:** ловим `isEcho`/чужой `authorId` в вебхуке Wazzup — продавец просто начал писать
клиенту, бот молчит. **Возврат:** продавец возвращает явной командой `#старт` (вводит там, где
клиент не видит — в Telegram-алерте, не в клиентском чате; пара: `#стоп`/автоглушение увести ↔
`#старт` вернуть) + авто-возврат после ~суток тишины, если ничего не висит — **кроме чувствительных
handoff (конфликт / деньги / агрессия): там только ручной `#старт`, авто-возврата нет.** Пока
молчит, бот всё равно записывает сообщения (без ответов); на `#старт` читает свежую карточку +
последние реплики и продолжает с контекстом.

## Guardrails, идемпотентность, аудит

- **Никогда без человека:** скидки/условия **вне конфига** (внутри конфига — бот сам, вкл. запись
  `discount`); рекомендации по здоровью/нагрузке ребёнка. Остальное — по хендофф-правилам.
  *(Ограничитель скидок — сам конфиг, он всегда на месте. Человек-проверка на оплате — съёмное
  колесо: уйдёт при авто-подтверждении через эквайринг, конфиг-ограничитель остаётся.)*
- **Идемпотентность роутера:** дедуп входящих по `messageId`, защита от ретраев канала
  (в старом боте ретраи Wazzup плодили дубли ответов — не повторять). **Дверь B** `messageId` не
  даёт (опрос Fitbase / вебхук Тильды) → свой ключ идемпотентности (`lead_id` + событие или флаг
  «уже коснулись»), чтобы повтор опроса / рестарт / двойной POST не сработали дважды.
- **Аудит:** логировать действия и решения бота (для разбора и доверия перед автономией).
- **Eval:** офлайн-набор тестовых диалогов прогонять перед сменой промпта/модели.
