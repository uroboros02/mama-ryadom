# Эквайринг и разноска оплат — Альфа + бот (рабочая записка)

> **Статус:** черновик-разбор (2026-06-23). Не финальное решение — фиксируем факты и развилку,
> чтобы не потерять. Связано с: [`fitbase-integration.md`](fitbase-integration.md) (🔴 в API
> Fitbase нельзя записать оплату/абонемент), [`tasks-open.md`](tasks-open.md) (задача по оплате),
> CRM-переоценкой (Мой Класс умеет это нативно).

## Корень проблемы (точный диагноз)

Это **не «бухгалтерия вообще», а реконсиляция платежей.**

- **Нативные платёжные шлюзы Fitbase:** только **CloudPayments, Payselection, TipTop** (Казахстан).
  Альфы среди них нет.
- **Автопосадка оплаты в ЛК Fitbase** работает **только** через вебхук от этих нативных шлюзов:
  шлюз сам генерит ссылку и сам говорит Fitbase «оплачено» → Fitbase сажает в карточку.
  Нет нативного шлюза → нет вебхука → нет автопосадки.
- 🔴 **В публичном API Fitbase нельзя записать ни платёж, ни абонемент** (`client-contract` —
  только чтение/удаление; метода «создать оплату» нет). То есть даже **нашим кодом** через
  официальный API «посадить деньги в ЛК» нельзя. Та же стена, что и с бронью.
- Вывод: ручная разноска — **дыра в системе**, а не недонастройка.

## Ключевой разлом проблемы (главный инсайт)

«Ручная разноска» = **две разные работы**, мы их склеивали:

1. **Детектив** — *кто* заплатил и *за что* (вот это ад на ~100 клиентах: деньги падают
   обезличенно, человек сопоставляет).
2. **Механика** — физически вписать оплату в карточку (один клик).

**Вебхук Альфы убивает работу №1.** Если ссылку на оплату генерит *наш* слой, мы при создании
ссылки зашиваем свой ярлык (`orderNumber = клиент Тимур, гимнастика, счёт №412`). Альфа
возвращает этот ярлык в callback → бот видит **«Тимур, гимнастика, 5000 ₽, оплачено 14:32»**,
уже опознанное, без детектива.

И это **ровно то, что боту нужно по нашей архитектуре**: на подтверждённой оплате бот делает
handoff продавцу («проверь оплату, дальше веди сам») + перевод лид→клиент. Боту **не нужно**
самому писать в Fitbase — ему нужно *знать*.

## Факты по Альфа интернет-эквайрингу (проверено 2026-06-23)

> ⚠️ Смотрели именно **интернет-эквайринг** (e-commerce / RBS-платформа), НЕ торговый
> (терминал) и НЕ СБП-кассу. Источники под `.../internet-acquiring/...` и `.../ecommerce/...`.

- ✅ **Банк шлёт callback (вебхук) на наш URL при событии оплаты.** События: успешный холд,
  **успешное списание**, возврат, отмена, отказ по таймауту.
- ✅ **Подпись есть.** Штатный вариант — с контрольной суммой; тип **симметричный** (callback-токен,
  по сути HMAC). Берём его, чтобы нельзя было подделать.
- ✅ **Наш `orderNumber`/`mdOrder` возвращается в уведомлении** → «кто за что» бот видит сразу.
- ✅ **Подключение — самому в ЛК:** `payment.alfabank.ru/lk` → Настройки → Callback-уведомления →
  включить, вписать URL, **статический** тип + **симметричная** подпись + сгенерировать токен.
  Запасной путь — письмо в `acquiring@alfabank.ru`.
- Механика регистрации заказа: `register.do` (одностадийный) / `registerPreAuth.do`
  (двухстадийный); статус — `getOrderStatusExtended.do`; адрес callback можно задать статически
  в ЛК или динамически при регистрации (`dynamicCallbackUrl`).

## 🔴 Открытый вопрос, который решает всё

Callback есть **только** в интернет-эквайринге. Поэтому всё зависит от того, **какой продукт
Альфы у центра сейчас стоит:**

| Что стоит | Вебхук об оплате | Годится для бота |
|---|---|---|
| **Интернет-эквайринг** (оплата по ссылке/онлайн) | ✅ callback | ✅ всё работает |
| **Торговый эквайринг** (терминал на ресепшене) | ❌ офлайн POS | ❌ бот оплату не увидит |
| **Перевод / QR / реквизиты** | зависит | отдельный разбор |

Моя схема (наш слой генерит ссылку → ловим callback) **требует интернет-эквайринга**. Если
сейчас платят картой на ресепшене (терминал) — это торговый эквайринг, callback там нет в
принципе; тогда либо подключать **дополнительно** интернет-эквайринг (лиды платят по ссылке от
бота), либо callback не вариант → сверка по выписке.

**Вопрос заказчику: как клиенты сейчас платят за абонементы?**

## Что остаётся нерешённым (честно)

- **«Посадка в карточку Fitbase»** вебхук НЕ чинит. Остаётся: (а) бот отдаёт админу готовую
  строчку «посади вот это» → один клик без детектива (уже не ад); (б) рискованный внутренний
  эндпоинт морды Fitbase (как с бронью, но это **деньги** — не любим); (в) на **Мой Класс**
  это нативно (`POST /payments` + платёжные вебхуки, поддержка Альфы подтверждена).
- Мета-вывод: ход с вебхуком Альфы **не спасает Fitbase целиком** — он ужимает боль (убирает
  детектив, закрывает потребность *бота*). Автопосадка в ЛК на Fitbase по-прежнему сама не
  делается → это всё ещё довод за Мой Класс.
