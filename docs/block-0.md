## Блок 0 — сводно (на одном экране)

**Задача Блока 0:** надёжно превратить любое входящее в **правильно опознанного лида с
сохранённой историей** — и либо припарковать на человека, либо (с F1) отдать мозгу и вернуть
ответ. Трубы и предохранители; «ум» — F1.

```
ВХОД — две двери:
  A. клиент написал (MAX/…)            B. заявка в CRM (Тильда/VK Ads/руками)
       ▼ push из канала                     ▼ детект: опрос Fitbase / вебхук Тильды
  [КАНАЛ] приём + перевод ↔ наш формат   [сигнал «новый лид»]
       └────────────────────┬────────────────────┘
                            ▼
      [РОУТЕР] дедуп messageId · опознать → ОДИН лид (найти до создания)
                            ▼
      [ВОРОТА] лид → дальше | действующий клиент → off (к админу)
                            ▼
      [РУЧНОЙ РЕЖИМ] человек за рулём? → молчим (#стоп / #старт)
                            ▼
      [ПАМЯТЬ] блокнот этого человека (один на лида, долгоживущий)
                            ▼
      [РЕШЕНИЕ] Блок 0 → эскалация | F1+ → мозг: ответить / эскалировать
                            ▼
      [ОТВЕТ] → через КАНАЛ; обмен пишем в ПАМЯТЬ

ПОД ВСЕМ: ОКНО FITBASE — единая дверь чтения/записи CRM (турникет ≤20 rps)
СБОКУ:    ЭСКАЛАЦИЯ — Task в Fitbase + TG-алерт кнопкой (#старт — возврат бота)
```

**Сквозные правила:** один человек = один лид (найти до создания) · читать карточку перед
ответом (Fitbase = правда, кэш = запаска) · бот сам выходит на контакт (каскад
MAX→WhatsApp→задача) · идемпотентность + турникет (свой баг страшнее трафика) · бот говорит
с F1 (в Блоке 0 клиенту не пишет) · память ходит за человеком, не за каналом.

## Режимы бота (state machine) — кто ведёт чат

Отдельно от **этапов воронки** (Неразобранные / В работе / Резерв / Архив — где лид в продаже, см. [`config.md`](config.md)): здесь — **кто за рулём чата**, бот или человек, и когда бот молчит. Консолидация из `escalation.md`, новых решений нет.

```
[ВНЕ ЗОНЫ / OFF] — действующий клиент (ворота по надёжному ключу: телефон/id канала;
       │           по имени НЕ сверяем; с нового номера не узнаём → ведём как лида).
       │           Бот молчит совсем → чат админу.
       ▼
[АВТО] — бот ведёт диалог (рабочий режим F1).
   │
   ├─► [РУЧНОЙ] — человек за рулём, бот молчит ПО ЭТОМУ ЛИДУ (с другими работает).
   │      Вход:  #стоп · продавец начал писать (isEcho / чужой автор) · автостоп-счётчик
   │      Молчит: входящие пишет в память, не отвечает
   │      Выход:  #старт · ИЛИ авто-возврат после ~суток тишины, если ничего не висит
   │
   └─► [ЧУВСТВИТЕЛЬНЫЙ HANDOFF] — конфликт / деньги / агрессия. Бот молчит.
          Вход:  эскалация по этим триггерам (мгновенно)
          Выход: ТОЛЬКО ручной #старт — авто-возврата НЕТ

[ОЖИДАНИЕ ОПЛАТЫ] — добавится на F3.
```

**Правила:**
- 🔴 `#старт` / `#стоп` вводятся **в Telegram-алерте, НЕ в клиентском чате**. Алерт = кнопка с опознавалкой (`#4232 Маша · оплата`) + контекст + подтверждение да/нет от случайного нажатия.
- Реализационно **РУЧНОЙ и ЧУВСТВИТЕЛЬНЫЙ — один режим** «человек за рулём» + флаг «без авто-возврата» для чувствительных.
- Автостоп-счётчик сработал → **РУЧНОЙ** (молчит) + алерт; возврат `#старт`.
- ⏳ Открыто: что считать «висит» для авто-возврата — **или** убрать авто-возврат и оставить только `#старт`.

## Блок 0 — компоненты (детально)

Блок 0 = трубы и предохранители, не мозг. Проверяется «на трубах» (вход опознан → отфильтрован
→ сохранён → при нужде задача продавцу); **общение с клиентом начинается с F1**. Порядок сборки:
окно Fitbase + канал MAX → роутер → ворота → эскалация + ручной режим → память.

**Шов 0→F1.** В Блоке 0 у бота нет заглушек-ответов клиенту (квитанций не шлём) — бот начинает
говорить на F1. Простое «упала заявка → задача продавцу» делает встроенная автоматизация Fitbase,
не наш бот. Записи Блока 0 (лид-подстраховка, заметка, задача) внутренние, клиент их не видит →
логируем, без воротец-ожидания; реальные воротца-одобрения — на F1 (бот пишет клиенту) и F3
(деньги). Кто подтверждает: поведение перед доверием — приёмка тобой по сценариям; рантайм-надзор
в обкатке — продавец.

**Роутер ≠ ЛЛМ.** Роутер — детерминированный код (дедуп, поиск, диспетчеризация): быстро,
предсказуемо, без токенов. ЛЛМ (Claude) — «мозг», его роутер вызывает на шаге «что ответить/
сделать» (F1+). Блок «РОУТЕР + ИИ-АГЕНТ» на схеме = две части: код-диспетчер + ЛЛМ-агент.

**1. Окно Fitbase** — единая дверь для всего чтения/записи в CRM (вся защита и контроль в одном месте).
- Операции в Блоке 0: найти лид · прочитать карточку · заметка · задача. **Создание лида — только
  подстраховка**, если карточки реально нет: опираемся на авто-создание канала/CRM (дверь B уже
  создаёт лид; Wazzup умеет на первое входящее). Правило: сначала найди, создавай только если точно не нашёл.
- **Дедуп слоёный:** CRM мёржит дубли карточек по телефону (`is_duplicate`) — опираемся на готовое;
  наше — `messageId` (не запустить бота дважды) и связка тредов «один человек» в момент ответа.
- Безопасность: токен в env, не в коде · перс. данные не размазывать по логам/лишним копиям
  (152-ФЗ), но скорость не жертвуем (переписка + кэш-запаска остаются) · необратимые записи
  логируем + идемпотентность · **турникет** = темп-ограничитель + откат на 429 (защита от нашего
  же бага, не от живого трафика) · Fitbase недоступен → падаем на кэш-запаску, бот не валится
  (на кэше — **позже, к F3**: ограничить необратимые/денежные действия, т.к. снимок может устареть).

**2. Роутер (вахта на входе)** — диспетчер на каждое входящее: решает путь, сам не отвечает.
- Обе двери (A — сообщение из канала, B — сигнал «лид появился») входят в одну точку.
- **Идемпотентность** по `messageId`: повтор вебхука молча выкинуть (старый баг — ретраи Wazzup
  плодили дубли ответов).
- **Опознать → один лид** (найти до создания).
- **Дирижёр станций:** ворота → ручной режим → память → решение; держит порядок и «СТОП»
  (ворота «действующий клиент» → бот off).
- Безопасность: проверять подлинность вебхука (подпись/секрет — дверь публичная) · идемпотентность
  · не плодить лиды · падать безопасно (ошибка → лог + эскалация).

**3. Канал (транспорт, MAX через Wazzup)** — труба наружу + переводчик формата.
- Слышать (вебхук) / говорить (Wazzup API) / **переводить** Wazzup-формат ↔ наш внутренний (мозг
  про вендора не знает → новый канал = новый переводчик, ядро не меняется). **Метим свои отправки**
  (для ручного режима).
- Дубли: **быстрый ack** (200 сразу, обработка асинхронно) + дедуп `messageId` — иначе ретраи
  Wazzup плодят дубли (старый баг).
- Безопасность: публичная дверь — проверять подпись/секрет (дверь живёт здесь, роутер получает
  уже проверенное) · ключ Wazzup в env · входящее — недоверенный ввод (вкл. prompt-injection,
  лечим на F1) · перс. данные не в логи.

**4. Краткосрочная память** — блокнот диалога, один **на человека** (якорь — телефон/карточка),
долгоживущий (переживает паузы в недели). Живёт у нас (Redis), не в Fitbase.
- Дописывает каждое сообщение (вкл. реплики, пока бот молчит); хранит между сессиями; отдаёт
  контекст (в Блоке 0 — держим наготове, пользуется мозг F1). Опирается на ключ-личность роутера.
- **Хранить всё, но в модель — только нужное** (свежий срез + сводка): весь архив = дорого/медленно
  (старый бот, prompt caching). Краткосрочная (транскрипт) ≠ долгосрочная (портрет/эпизоды — F1+).
  Ключевые факты дублируем в Fitbase `Note`.
- Безопасность: **главный склад перс. данных** (152-ФЗ: срок хранения, сервер в РФ, доступ) · не
  терять историю на сбое (персистентность/бэкап) · память верна настолько, насколько верен ключ
  (рискует расколоть/склеить людей — как у ворот).

**Перед сборкой — что добыть/подтвердить (ворота к коду):**

*Доступы / ключи (добыть):*
- Fitbase: API-токен + `domain` + тестовый домен-песочница.
- Wazzup: аккаунт/ключ + подключённый MAX-канал + тестовый канал.
- Telegram: бот для алертов + chat id продавца. Redis. Anthropic-ключ (уже для F1).

*Fitbase — подтвердить по API:*
- Формат входа (Bearer + `domain`), срок жизни токена.
- Поиск лида по телефону / id канала; реальные поля каналов (`phone, max, vk, …`).
- Как отличить действующего клиента (активный `ClientContract`).
- Фильтр по `updated_at` (радар / опрос двери B).
- Поля задачи (`responsible` = id продавца — узнать; `available_to`, `description`), `Note`.
- Что/как авто-создаёт связка Wazzup→Fitbase (чтоб «создание лида = подстраховка»).
- Что делает сам с дублями (`is_duplicate`). Заодно: есть ли вебхук на оплату (для F3).

*Wazzup / MAX — подтверждено по докам (2026-05-29), осталось проверить на живом payload:*
- Вебхук Wazzup отдаёт: `messageId` (uuid), `chatType` (вкл. `max`), `chatId`, `contact.phone`
  («для Telegram и MAX»), `text`, `contentUri`, `type`. ⚠️ Телефон для MAX — **проверить на реальном
  входящем** (MAX сам отдаёт номер только если юзер «поделился контактом»; вероятно Wazzup подтягивает,
  но не гарантия). Опознавалка — **пул** (`chatId`/MAX `user_id` + имя есть всегда; телефон когда есть).
- Отправка: по `chatId` **или `phone`/`username` для MAX** → дверь B (писать по номеру) работает.
- Подлинность вебхука: заголовок `Authorization: Bearer <crmKey>` (общий секрет, мы задаём; HMAC нет).
- Реплики продавца / метим свои: поле `isEcho` (+ `authorId`/`authorName`) — закрыто.
- **Право писать первым зависит от тарифа Wazzup** (на «Inbox» нельзя; «Старт» — есть, но лимит ~50
  диалогов/мес → мало): нужен тир с API + write-first без узкого лимита (**Pro/Max**), уточнить лимит
  первых сообщений/мес при подключении.
- Тильда → наш вебхук вторым адресом (лендинг твой — настроить).
