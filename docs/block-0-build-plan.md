## Зачем этот файл

`block-0.md` отвечает **что** строим и **как поймём, что готово** (приёмочный чек-лист).
Этот файл — **как и в каком порядке** собираем Блок 0 **на заглушке**, чтобы строить и
тестировать поведение **off-clock** (часы триала Fitbase не тикают) и **без 152-ФЗ** (данные
ненастоящие). Каждый шаг закрывает группу приёмочного чек-листа.

## Что значит «на заглушке» (Фаза А)

Блок 0 — трубы и предохранители, клиенту он **не пишет** (общение начинается с F1). Проверяем
**по приборам** (логи решений + состояние в Redis/карточке), не по диалогу.

| Компонент | В Фазе А |
|---|---|
| Fitbase (CRM) | **mock** в памяти процесса (по контракту `fitbase-integration.md`) |
| Wazzup / MAX | **mock**: вбрасываем фейковые вебхуки, ловим, что бот «сделал» |
| TG-алерт продавцу | **mock**: «коробка алертов» |
| Redis (память/дедуп) | **по-настоящему** (локально / `fakeredis`) — память обязана пережить рестарт |
| Claude (мозг) | **не нужен** — Блок 0 не думает (мозг — F1). Можно строить до пополнения Anthropic |

**Честная оговорка:** заглушка ≠ жизнь. В триале (Фаза Б) вылезут сюрпризы там, где контракт
неизвестен (телефон в MAX, поиск лида, POST-бронь — см. `fitbase-integration.md`). Заглушка
закрывает плотную предсказуемую часть → в триал входим «доводкой», а не «стройкой».

## Чего в Блоке 0 НЕ делаем (анти-усложнение)

- Не пишем клиенту, не зовём Claude, не делаем настоящую бронь/оплату (F1/F3).
- Не строим очередь задач (Celery и пр.) — «быстрый ack + обработка фоном» = встроенный механизм FastAPI.
- Не делаем «плагин-систему каналов» — один переводчик MAX; новый канал добавим, когда понадобится.
- Не пишем защитный код под невозможные случаи.

## Стек (утв. 2026-06-02)

- **Python + FastAPI** — вебхук, быстрый ack, асинхронный радар; встроенная валидация входящего.
- **Redis** локально; в тестах — `fakeredis`. Railway-Redis подключим, когда удобно.
- `httpx` (HTTP к Fitbase), `pytest` (поведенческие тесты = приёмочный чек-лист).
- Часовой пояс **Уфа (UTC+5)** — одним местом в конфиге.

## План по шагам

Строим по одному: код шага → проверка поведения → «добро» → коммит → следующий. Формат:
**что · проверка (поведение) · что сломается, если неверно.** В скобках — группа приёмочного
чек-листа из `block-0.md`.

- [x] **Шаг 0 — Каркас + моки (рельсы).** ✅ 2026-06-02
  Структура, зависимости, `config.py` (env, TZ Уфа), Redis, пустые моки, `/health`.
  · *Проверка:* сервер стартует, `/health`=200, тест вбрасывает вебхук → 200. — **зелёная** (4 теста + живой uvicorn).
  · *Риск:* переусложнить каркас — держим минимальным.

- [x] **Шаг 1 — Канал MAX: приём вебхука.** (гр. 1 + часть гр. 4) ✅ 2026-06-02
  Проверка подписи (`Bearer <crmKey>`) → **быстрый ack 200** → обработка фоном; перевод формата; метка `isEcho`/чужой `authorId`.
  · *Проверка:* поддельный вебхук отбит; валидный → 200 сразу; эхо/продавец → помечено. — **зелёная** (12 тестов + живой uvicorn).
  · *Учтён конверт Wazzup:* тест-пинг `{test:true}`→200 (hard-gate подключения), пачка `messages[]`, игнор `statuses`. Внутренности сообщения (`phone` в MAX, поля автора) — проверить на живом payload (Фаза Б).
  · *Если неверно:* публичная дверь нараспашку; без ack ретраи плодят дубли.

- [x] **Шаг 2 — Идемпотентность по `messageId`.** (гр. 1 🔴) ✅ 2026-06-02
  Store «видели messageId» в Redis; повтор молча выкинуть.
  · *Проверка:* один вебхук дважды → обработан один раз. — **зелёная** (16 тестов; полный HTTP-путь на fakeredis).
  · *Реализация:* `app/router.already_processed()` = атомарный `SET NX EX` (TTL сутки); `app/redis_client.py` — общая точка Redis; `tests/conftest.py` — fakeredis всем тестам. Живой прогон с настоящим Redis — когда поднимем Redis (Railway/локально-в-папке).
  · *Если неверно:* двойные действия на каждый ретрай.

- [x] **Шаг 3 — Окно Fitbase (на mock).** (гр. 5 🔴) ✅ 2026-06-02
  Единая дверь: найти/читать/заметка/задача. **Турникет ≤20 rps + бэк-офф 429**, токен из env, падение Fitbase → кэш-запаска (бот жив), необратимые записи логируются + идемпотентны.
  · *Проверка:* Fitbase «лежит» → бот не падает (лог+эскалация); >20 rps → тормозит; задача/заметка — один раз + в лог. — **зелёная** (24 теста, фейковые часы).
  · *Реализация:* `app/fitbase.py` (окно + `RateLimiter` + бэк-офф + кэш-запаска + идемпотентная запись через claim/release); `app/mocks/fitbase_mock.py` (in-memory CRM с `down`/`fail_429_times`); `alert_mock` (коробка эскалаций). Заметка лида → `custom_fields` (Note привязан к клиенту). **Окно ещё не вплетено в поток вебхука — это Шаг 4 (ворота).**
  · *Если неверно:* свой баг забивает CRM; падение CRM роняет бота.

- [x] **Шаг 4 — Роутер + Ворота.** (гр. 2 и 3 🔴) ✅ 2026-06-02
  Опознать → **один лид** (искать до создания). Ворота: активный `ClientContract` по надёжному ключу (телефон/id канала) → бот **off**, чат админу. Имя — слабая подсказка.
  · *Проверка:* один человек с двух каналов → один лид; действующий клиент → off; тёзки не склеиваются; новый номер → как лид. — **зелёная** (30 тестов + живой поток /webhook).
  · *Реализация:* `app/router.route()` (ворота → свой Redis-индекс `idx:канал:значение→lead_id` → создать подстраховкой); `app/fitbase` +`active_client_by_contact` (Fitbase лёг→считаем лидом) +`create_lead`; `main` — провайдер окна + маршрутизация (эхо/продавец пропускаются); `channel_max` +`name`. **Дев-режим `REDIS_URL=fakeredis`** (Redis в процессе, для живых демо без установки).
  · *Край:* дедуп `messageId` под строго одновременными ретраями+in-process fakeredis может прогонять (на настоящем Redis `SET NX` атомарен; и есть 2-й рубеж — идемпотентность записи → лид создаётся 1 раз). Авто-склейка двух каналов без общего ключа — известный край (отложен).
  · *Если неверно:* дубли/склейка людей; глушим живого лида.

- [ ] **Шаг 5 — Режимы / стоп-кран.** (гр. 4 🔴)
  State-machine на лида: АВТО/РУЧНОЙ/ЧУВСТВ; `#стоп`/`#старт` (моделируем приход из TG-алерта, не из чата клиента); эхо/продавец → РУЧНОЙ; автостоп-счётчик (>N сообщений/мин) → РУЧНОЙ + алерт.
  · ✅ **Решено 2026-06-02:** на старте **только ручной `#старт`**, авто-возврат по тишине отложен (вернёмся, если понадобится). Чувствительный handoff — тоже только `#старт` (как и было).
  · *Проверка:* `#стоп` глушит этот лид, по другим работает; автостоп срабатывает; состояние переживает рестарт.
  · *Если неверно:* бот «строчит сам с собой» или глушит не того.

- [ ] **Шаг 6 — Память (блокнот за человеком).** (гр. 7 🔴)
  Один блокнот на лида (ключ — личность, не канал), Redis; дописывает каждое входящее (в т.ч. пока молчит); переживает рестарт. Срез+сводка для модели — это F1.
  · *Проверка:* память за человеком, не за каналом; рестарт → история цела.
  · *Если неверно:* теряем контекст / путаем людей.

- [ ] **Шаг 7 — Радар двери B (опрос Fitbase, на mock).** (гр. 6 🔴)
  Инкрементальный опрос по `updated_at`; **watermark «докуда дочитал» переживает рестарт**; overlap-окно + дедуп событий.
  · *Проверка:* повторный опрос/рестарт → без дублей; новый/обновлённый лид в окне виден.
  · *Если неверно:* пропуск заявок / повторные касания на рестарте.

- [ ] **Шаг 8 — Время Уфа + шов 0→F1 + полный прогон.** (гр. 8 🔴 + финал)
  Единый TZ Уфа во всех таймерах; подтверждаем — клиенту не пишем (записи внутренние, логируются). **Прогон всего приёмочного чек-листа на искусственных входящих.**
  · *Проверка:* зелёный весь чек-лист «на трубах».
  · *Гейт:* «Блок 0 держит → можно цеплять F1».
