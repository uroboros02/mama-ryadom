# Fitbase — контракт интеграции (API v2)

> Что и как бот читает/пишет в Fitbase. Собрано из OpenAPI-спеки v2 (2026-06-02):
> Swagger UI `https://api.fitbase.io/site/docs_v2`, сырая спека `https://api.fitbase.io/api_v2`.
> Имена полей — по спеке; помеченное «проверить в триал» уточняем на живой CRM.
>
> **Статус волны 1:** бронь и оплата — продавец руками (human-in-the-loop, принцип 12).
> 🔴 **Автобронь (`POST /schedule-registration`) — MUST-SOLVE (100%) для автономии** — это суть
> продукта (бот бронирует сам). Спрошено у разработчиков Fitbase 2026-06-02, ждём ответ.

## Авторизация
- **Bearer-токен** (JWT). Создаётся в Fitbase: *Настройки → Интеграции → ФитБейс API*, привязан к выбранным клубам.
- Заголовки каждого запроса: `Authorization: Bearer <токен>` + `domain: <поддомен клуба>`.
- База: `https://api.fitbase.io/api/v2/`.

## Что умеет API (по спеке v2) и зачем нам
| Эндпоинт | Методы | Зачем нам |
|---|---|---|
| `/lead`, `/lead/{id}` | GET, POST, PATCH | карточка лида: создать, читать, двигать этап/факты |
| `/contact`, `/contact/{id}` | GET (`?contact=`), POST, PATCH | поиск клиента по контакту (гейт «активный клиент») |
| `/client`, `/client/{id}` | GET, POST, PATCH | действующий клиент |
| `/client-contract`, `/{id}` | **только GET, DELETE** | проверка активного абонемента (гейт Блока 0) |
| `/purchase-funnel(-step)` | **только GET** | этапы воронки (id шагов) |
| `/task`, `/task/{id}` | GET, POST, PATCH | задача продавцу + алерт (handoff) |
| `/note`, `/note/{id}` | GET, POST, PATCH | заметки (⚠️ привязка к клиенту, не лиду) |
| `/schedule`, `/schedule/{id}` | **только GET** | расписание, вместимость (`max_register`) |
| `/schedule-registration`, `/{id}` | **только GET, DELETE** | читать брони (создавать — нельзя, см. «Ограничения») |

## Ключевые поля
- **Lead:** `name, surname, phone, email, vk, telegram, max, client_id, funnel_step, custom_fields, is_duplicate, tags, advertising_source, utm_*, manager, description`.
- **Client:** `name, surname, birth_date, status, contacts, custom_fields, is_archive, manager`.
- **ClientContract:** `client_id, begin_date, end_date, closed, visits_left, price` → «активный» = не `closed` / `end_date` в будущем / `visits_left > 0`.
- **Task:** `lead_id, client_id, status, description, responsible, available_to, category`.
- **ScheduleRegistration:** `client_id, event, status, auto_delete_time`.
- **Schedule:** `training, place, age_type, date, time_start, max_register, stop_registration`.

## Как наша модель ложится на Fitbase
- Лид бота → **Lead**. Каналы → `max/telegram/vk/phone`. Факты квалификации → `custom_fields`. Этап → `funnel_step`.
- Действующий клиент → **Client** + активный **ClientContract** (гейт Блока 0 «лид vs клиент»).
- Handoff/алерт продавцу → **Task** (`lead_id` + `responsible` = продавец + `available_to` = срок).
- `note_fact` (факт-память лида) → **`custom_fields` лида** (Note привязан к клиенту). *Проверить в триал: примет ли `POST /note` поле `lead_id`.*
- Свободный слот → `schedule.max_register` − число активных `schedule-registration` по событию.

## Ограничения (важно)
- 🔴 **MUST-SOLVE (100%): бронь не создаётся через v2 API** (`schedule-registration` — нет POST).
  - Волна 1: на пробную записывает **продавец руками** (наша human-in-the-loop рамка, принцип 12).
  - Для автономии (суть продукта — бот бронирует сам) решаем: (1) 🥇 **официальный способ от Fitbase** — спрошено у разработчиков 2026-06-02, ждём; (2) 🥈 запасной — DevTools-разведка внутреннего эндпоинта морды на триале (осторожно: деньги-смежное, недокументированное, может ходить по сессии-куке). Трекинг — [`tasks-open.md`](tasks-open.md) §F3.
- 🔴 **Абонемент/договор не создаётся** (`client-contract` — только чтение). Оплата/оформление — не через этот API (эквайринг отдельно, F3).
- ⚠️ **Нет прямого поиска лида по телефону/каналу** (фильтры `/lead`: этап, менеджер, `updated_at`). Предпочтительно — нативный поиск в Fitbase (спрошено у разработчиков 2026-06-02); пока работаем на **своём индексе** (Redis: канал→`lead_id`) + id входящего из Wazzup. DevTools на триале может показать search-эндпоинт морды.

## Опрос («радар», вместо вебхуков — их нет)
- У всех списков есть `?updated_at` + `page`/`page_size` → инкрементальный опрос изменений.
- Курсор/watermark по `updated_at`, overlap-окно, дедуп событий (детали — Блок 0).
- Лимит **20 rps** (подтверждён ранее) → бэк-офф на 429.
