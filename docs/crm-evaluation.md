## Почему пересматриваем — триггер

Две боли Fitbase, обе про деньги/автономию:
1. 🔴 **Бронь через API — НЕВОЗМОЖНА.** В v2 API метода нет (только GET/DELETE). **Вендор подтвердил
   2026-06-23:** официального способа записать клиента на занятие со стороны интеграции нет —
   есть только эндпоинты **клиентского кабинета под токеном клиента** (клиент сам входит по
   SMS/звонку и записывается). Бот так автономно бронировать не может. Это **суть продукта**, поэтому
   Fitbase как фундамент автономной версии отпадает.
2. **Эквайринг:** у Fitbase нативно только CloudPayments/Payselection (~2.7%), **нет Альфы@1.4%**;
   подключение Альфы = ручная проводка оплат (нет автосверки), т.к. API не пишет платежи/договоры.

> Плюс Fitbase (честно): есть **готовый коннектор 1С:Бухгалтерия** — по бухгалтерии он как раз неплох.

## Мой Класс — разбор (самый проверенный)

Источник: их OpenAPI (145 эндпоинтов) + **прямые ответы поддержки 2026-06-23** (тех. отдел Вадим + продажи).

**Закрыто / подтверждено:**
- 🟢 **Бронь ботом серверно** — `POST /v1/company/lessonRecords` (`userId`+`lessonId`, флаг `test`=пробное,
  `free`=бесплатно), под ключом компании, **без участия сотрудника** (подтвердил их API-специалист).
- 🟢 **Поиск по телефону** — `GET /users?phone=` (даже по части номера) → дедуп, гейт «лид/клиент».
- 🟢 **Задачи / заметки / воронка** — `POST /tasks`, `userComments`, статусы клиента (двигаются по API).
- 🟢 **Эквайринг Альфа = нативно + свой мерчант** (свои 1.4%, МК сверху 0%), оплата автопроводится в
  карточку по подписанному callback. **Ссылку на оплату можно отправить БЕЗ входа клиента в кабинет**
  (уникальная, сбрасываемая) → снимает то трение, из-за которого отвергли путь Fitbase.
- 🟢 **Вебхуки есть** (события, настраиваются сценарием в CRM; список — их статья 81).
- 🟢 **Данные в РФ** (ЦОД; см. ниже про расхождение) → 152-ФЗ residency; **согласие собирается прямо
  в онлайн-записи** (галочка + ссылка на наш текст, статья 660); есть «соглашение об обработке» (DPA).
- 🟢 **Модель «мама↔ребёнок»:** ребёнок = карточка (ученик), родитель = **признак** (доп. поле) в карточке
  ребёнка; **телефон — ответственного родителя**. Фича «Семья» — НЕ для нас (она для тех, кто сам ходит и
  делит абонемент). → по каждому ребёнку своя карточка/сделка.
- **Тариф:** API доступен на **Pro**; цена = тариф + число активных групп (дадут КП, если назвать число).

**Затыки (наша сторона / операционка):**
- 🔴 **Занятия не создаются наперёд** — их заводит сотрудник вручную на срок (или мы через `POST /lessons`).
  Иначе боту не во что записывать. → дисциплина «расписание заведено вперёд» либо наш код.
- **Нет поиска по chatId канала** + **`wazzupAssigns` мы сами не пишем** (только их интеграция/интерфейс)
  → держим **свой индекс chatId→ученик** (особенно MAX = 60%, там телефона в вебхуке нет).
- **Нет идемпотентности** → дедуп у нас; ссылка на оплату не сбрасывается → возможна двойная оплата (сверять).
- **Лимит 7 rps** → ошибка, при злоупотреблении блок IP; вебхук сам отключится, если наш приёмник 10 мин в таймаутах.
- ⚠️ **Слабое место — 1С:** готового коробочного коннектора нет (в отличие от Fitbase) — только экспорт
  из «Финансы» + интеграция по API / их платная доработка «Link». Уточнить, что реально нужно бухгалтеру.
- ⚠️ Хостинг назвали по-разному: продажи — **Netangels** (IP `5.181.253.117`), тех — **Ростелеком ЦОД Екб**.
  Оба РФ (residency ок), уточнить точную площадку для бумаг.

## 1С:Фитнес клуб — разбор

Источник: их API V3 (Apiary blueprint, `app.1c.fitness/fitnessapi`). Вопросы технарям заданы, **ждём ответ**.

**Сильное:**
- 🟢 **Нативный 1С** — закрывает бухгалтерию (главное слабое место Мой Класс).
- 🟢 **Встроенные согласия** (обработка ПД / оферта / политика) — прямо под 152-ФЗ.
- 🟢 Эквайринг **Альфа** (`use_bank=alfa`) + Сбер/Тинькофф/ЮKassa/СБП + `payment_link`; рекуррент/привязка карт.
- 🟢 `available_slots` (свободные места), `relatives` (родственники — чтение), `token-master` (родитель платит за ребёнка),
  лист ожидания, native статус брони `temporarily_reserved_need_payment` (возможно авто-холд — проверить).
- 🟢 Бронь ботом **достижима**, но через **«танец токенов»**: `GET /pass_token?phone&sign`
  (sign=SHA256 `phone:..;key:СЕКРЕТ`) → `reg_and_auth_client` (выдаёт `user_token`, создаёт клиента) →
  `client_to_class`. Серверно, без SMS клиента — **если у нас есть секретный ключ** (уточнить).

**Затыки:**
- 🔴 **API клиент-центричный** — каждое действие за клиента нужен `user_token` (а не чистый «admin + client_id»).
  Сложнее и хрупче Мой Класс; зависит от секретного ключа и жизненного цикла токена.
- 🔴 **Нет API задач**, **нет API заметок**, **нет поля воронки/этапа** (у лида только телефон/имя/коммент/utm + теги).
  → задачи = только алерт у нас; заметки → доп. поля/теги; воронку ведёт бот.
- ⚠️ Поиск клиента — через авторизационный флоу, не чистым админским GET.
- ⚠️ Привязка канала (chatId) — нет → наш индекс. Создание связи «родственник» по API — не подтверждено. Вебхуки/лимиты в спеке не описаны.

## MobiFitness — ВЫБРАН организацией (2026-06-23) · что это значит для бота

- **MobiFitness ≠ 1С:Фитнес клуб** (в старых заметках были склеены — неверно).
- Это **своя платформа** (встроенная CRM + приложения), которая **может** работать и как
  **приложение поверх** внешнего бэкенда (1С:Фитнес или ClubIS).
- 🟢 Почему бизнес мог выбрать: у их **собственной CRM — лучшая воронка продаж** (стадии, доска сделок,
  задачи, заметки) + брендированное **приложение для мам**. Это сильные продуктовые стороны.
- 🔴 **Но для интеграции бота — главные риски (по ответу вендора 22.06):**
  - API **закрытое/приватное**, доступ дают **только «в порядке исключения»** (SaaS-first; используется «как
    есть», под нас не дорабатывают). → **Сначала надо ДОБЫТЬ доступ.**
  - API **отдаёт только то, что видно в мобильном приложении — без сводок/выгрузок по базе.** Если это так
    буквально — нет запросов «по всей базе» → **ломаются поиск лида по телефону, дедуп и опрос на входе**
    (инвариант «один человек = один лид»). **Это потенциальная стена — проверить ПЕРВЫМ.**
  - Один плюс: токен можно выдать **не привязанным к клиенту** (чуть больше серверной гибкости).
- **Что делаем (не «отпал», а «проверяем под решение»):**
  1. Получить **реальный доступ к API** (ключ/доку) — даже если «по исключению», объяснить кейс.
  2. Прогнать **эмпирический чек-лист** ниже (как сделали с Мой Классом) — особенно п.0 «есть ли запросы по базе».
  3. Если запросов по базе НЕТ → **архитектурный запасной план:** бот держит **свой стор лидов** (Redis/наша БД)
     как источник правды по лидам, а MobiFitness получает только **записи** (что API позволяет писать) +
     брендированное приложение как витрина. Тогда инвариант «CRM = источник правды» для **лидов** смещается
     к нам (для действующих клиентов источник правды — всё равно CRM/админ).

## Олимпико — разбор (`lk.ufagymnastics.ru`, легаси старого бота)

> Всплыло как идея «вернуться к старому боту Олимпико и доработать» (2026-07-06). Источник —
> рабочий код старого бота: `~/Claude/olympico-bot (old-sportcrm)` (`main.py`, `schedule.py`,
> `old-bot-architecture.md`) + ветка `legacy-olympico-full-history` в этом репо. Своя sport-CRM
> Олимпико/ufagymnastics. **Прогнали по нашему чек-листу 0–10 — ниже.**

**Как устроен доступ (в этом вся суть — два типа):**
- 🟢 **Открытые хуки `/hook/*`** (без авторизации, только `x-www-form-urlencoded`, JSON не принимает):
  - `/hook/idLeadReserve` — читать свободные слоты (squad, недели);
  - `/hook/dbedit?table=idLead` — **создать** лида (`oper=add`) и **редактировать** (`oper=edit`):
    имя/телефон/info/source/squad, **сменить статус**, проставить дату+тренера.
- 🔴 **Закрытый `/rows`** (jqGrid-поиск: фильтр по полю, пагинация, сортировка) — **требует логина**.
  `apiKey` его не открыл (Bearer/header/cookie/param → Forbidden).

**Статусы лида** (`idLead.status`): `c` неразобранное (дефолт) · `w` в работе · `i` на пробное ·
`d` пришёл (ставит менеджер вручную) · `a` зачислен · `r` отказ · `z` резерв · `x` архив.

**Прогон по чек-листу 0–10:**

| # | Нужно боту | Олимпико | Вердикт |
|---|---|---|---|
| 0 | Запросы по базе (список/поиск/изменённые) | `/rows` есть, но **под логином**; apiKey не открыл | 🔴 **СТЕНА** |
| 1 | Найти лида по телефону (дедуп) | код есть (`find_lead_by_phone`), упирается в `/rows` | 🔴 блок |
| 2 | Создать/обновить лида (+возраст/источник) | `/hook/dbedit`, partial-edit (read-modify-write) | 🟢 |
| 3 | Лид vs действующий клиент (читать абонемент) | чтение карточки/членства = через `/rows` | 🔴 блок |
| 4 | Воронка — двигать / читать этап | **двигать** `status` ✅; **читать** текущий = `/rows` | 🟡 пишем, не читаем |
| 5 | Бронь пробной (создать/отменить/перенести) | `book_slot` пишет дату+тренера, **вживую НЕ тестили** (`CRM_ENABLED=false`); «слот» = штамп даты на карточке, без контроля вместимости | ⚠️ непроверено |
| 6 | Посещение/неприход (догрев F4/F5) | статус `d` ставит менеджер **вручную**; читать = `/rows` | 🔴 блок |
| 7 | Задачи продавцу | нативных нет; алерт = наш (мессенджер) | 🔴 нет |
| 8 | Оплата (факт + ссылка без логина + Альфа) | **ничего**; платная пробная = клиент шлёт скрин перевода | 🔴 нет |
| 9 | Абонемент — читать активный / выдать | не подтверждено, тот же барьер `/rows` | 🔴 блок |
| 10 | События — вебхук/опрос | CRM нас **не** уведомляет; опрос «изменённые» = `/rows` (закрыт) | 🔴 нет сигнала |

**Где упрёмся (честно):**
- 🔴 **Главная стена — та же, что у Fitbase/MobiFitness, только с другой стороны:** открыта **запись**,
  закрыто **чтение по базе** (`/rows`). Без него нет поиска по телефону → **дедупа**. А CRM сама создаёт
  карточку на каждое входящее WhatsApp → бот без поиска **плодит дубли** (именно поэтому в проде держали
  `CRM_ENABLED=false` — бот по факту в CRM не писал).
- 🔴 **Целые блоки роудмапа отсутствуют как класс:** оплаты/эквайринг (F6), нативные задачи,
  выдача/чтение абонемента. Платежи — вручную скринами.
- ⚠️ **Бронь никогда не тестили вживую**; «бронь» = штамп даты на карточке, вместимость не контролируется.
- 🔴 **Стратегически:** по нашему `CLAUDE.md` эта CRM **выводится из эксплуатации** — возврат = ставка
  на систему, которую бизнес сворачивает.

**Единственный реальный актив — рабочий код** (Wazzup, Redis-память, дедуп по `messageId`, асинхронный
вебхук, prompt caching, стоп-кран). Он **переносится в любой вариант** независимо от CRM.

**Вердикт:**
- как **быстрый тактический перезапуск до-КЭВ части** (квалификация + запись на пробную) — **возможно,
  но при одном жёстком условии:** срмщики откроют хук «найти лид по телефону» (их и ждали). Без этого
  автозапись = фабрика дублей.
- как **фундамент полной автономии по роудмапу** — **нет** (read-стена + ноль оплат/абонементов/задач
  + CRM на списании). Слабее даже Мой Класса (тот прошёл весь чек-лист вживую).

## 🔬 Эмпирический чек-лист интеграции CRM (выводы из живой проверки Мой Класса 2026-06-23)

> Мы НЕ поверили на слово — взяли триал Мой Класса и **прогнали по живому API весь роадмеп бота**
> (страховка от «стены Fitbase» в середине разработки). Ниже — **что должен уметь API любой CRM под
> нашего бота** (проверять на триале ДО постройки) и **универсальные грабли**, которые нашли.
> **Это рабочий инструмент проверки MobiFitness.**

### Что бот должен уметь через API (проверить эмпирически)
0. 🔴 **(для MobiFitness — ПЕРВЫМ) Запросы/выборки по базе** — список лидов, фильтр по телефону,
   «изменённые с даты». Без этого нет дедупа/опроса. На Мой Классе ✅; у MobiFitness под вопросом.
1. Найти лида **по телефону** (точно; см. грабли про нормализацию) — дедуп.
2. Создать / обновить лида (+ доп-поля под квалификацию: **ДР/возраст ребёнка**, источник).
3. **Лид vs действующий клиент** — прочитать активный абонемент/договор (гейт «бот off для клиента»).
4. **Воронка** — прочитать и **двигать этап**.
5. **Бронь пробной** — создать запись в слот; **отменить**; **перенести**.
6. **Посещение / неприход** — прочитать (пришёл/пропустил) для догрева F4/F5.
7. **Задачи** продавцу — создать + прочитать очередь (handoff).
8. **Оплата** — увидеть факт (вебхук/опрос) + **ссылка на оплату без логина клиента** + Альфа-эквайринг.
9. **Абонемент** — прочитать активный (гейт) + **выдать** (граница лид→клиент).
10. **События** — вебхук на наш URL ИЛИ опрос «изменённые с даты».

### Универсальные грабли (нашли на Мой Классе — проверить/заложить на ЛЮБОЙ CRM)
- 🔴 **Телефон в поиске может НЕ нормализоваться** — `79…` находит, `89…`/`+7…`/с пробелами нет. → бот
  **сам нормализует** телефон к единому виду перед поиском, иначе плодит дубли.
- 🔴 **Телефон НЕ уникален** — двое детей на номер мамы = две карточки в выдаче. → дедуп обрабатывает
  **>1 совпадения**, не «телефон есть = тот же человек».
- ⚠️ **Обновление может быть ПОЛНОЙ ЗАМЕНОЙ** (пропустил поле → обнулил). → бот всегда **read-modify-write**.
- ⚠️ **Идемпотентности может не быть** (двойной POST = два) → свой дедуп по `messageId`; **ссылка на оплату
  может оплатиться дважды** → сверять.
- ⚠️ **Вебхуки:** доставка ≠ подписка на события (может требовать «сценарий»/выбор событий); часто **без
  подписи** → секретный URL + **перепроверка денег через API** (наш инвариант); могут настраиваться **только
  в UI**. Если push сложен — **опрос** закрывает (он и так в плане). На Мой Классе: тест-вебхук доходит, на
  реальные события сам по себе нет, действия «отправить webhook» у сценария нет → остаётся опрос/задача.
- ⚠️ **DELETE может не работать** (отдавал 500) → отмена/снятие **через смену статуса**, не удаление.
- ⚠️ **Расписание/занятия = настройка сотрудника в UI**; бот **записывает в готовые слоты**. Структуру
  (курсы/группы) через API создать может быть нельзя (у МК курсы — `Parse error`, только UI).
- ⚠️ **Лимит rps** (у МК 7/сек, при превышении бан IP) → наш «турникет»-троттл.
- ⚠️ **Онбординг в клиентское/родительское приложение ≠ автомат.** Включение кабинета письмо НЕ
  шлёт; клиент получает доступ либо сам (вход → «восстановить пароль»), либо сотрудник жмёт
  «сбросить/отправить пароль» в карточке. Логин = **email**, приложение для клиентов отдельное
  (у МК — «Твой Класс», отправитель `lk-noreply@tvoyklass.com`). → это **трение на пути мамы**.
  🔴 **Важно для MobiFitness:** его главный плюс — брендированное приложение для мам, поэтому
  проверить **первый вход мамы** (автоинвайт? сама ставит пароль? логин телефон/email? насколько
  это гладко) — иначе козырь продукта упирается в онбординг-барьер.

### Эталон: что Мой Класс **подтвердил вживую** (ориентир «как должно быть»)
На живом триале end-to-end прошло: поиск по телефону, создать/обновить лида, вход + движение по воронке,
создать занятие, **записать на пробную**, прочитать + проставить посещение (с авто-списанием с абонемента),
задача, выдать абонемент, прочитать активный, создать платёж, **ссылка на оплату без логина**. **Стены нет.**
Для бота это был бы чистый фундамент — берём как **мерку для MobiFitness**.
