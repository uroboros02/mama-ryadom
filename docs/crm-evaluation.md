## Почему пересматриваем — триггер

Две боли Fitbase, обе про деньги/автономию:
1. 🔴 **Бронь через API — НЕВОЗМОЖНА.** В v2 API метода нет (только GET/DELETE). **Вендор подтвердил
   2026-06-23:** официального способа записать клиента на занятие со стороны интеграции нет —
   есть только эндпоинты **клиентского кабинета под токеном клиента** (клиент сам входит по
   SMS/звонку и записывается). Бот так автономно бронировать не может. Это **суть продукта**, поэтому
   Fitbase как фундамент автономной версии отпадает.
2. **Эквайринг:** у Fitbase нативно только CloudPayments/Payselection (~2.7%), **нет Альфы@1.4%**;
   подключение Альфы = ручная проводка оплат (нет автосверки), т.к. API не пишет платежи/договоры.

> Плюс Fitbase (честно): есть **готовый коннектор 1С:Бухгалтерия** — по бухгалтерии он как раз неплох.

## Мой Класс — разбор (самый проверенный)

Источник: их OpenAPI (145 эндпоинтов) + **прямые ответы поддержки 2026-06-23** (тех. отдел Вадим + продажи).

**Закрыто / подтверждено:**
- 🟢 **Бронь ботом серверно** — `POST /v1/company/lessonRecords` (`userId`+`lessonId`, флаг `test`=пробное,
  `free`=бесплатно), под ключом компании, **без участия сотрудника** (подтвердил их API-специалист).
- 🟢 **Поиск по телефону** — `GET /users?phone=` (даже по части номера) → дедуп, гейт «лид/клиент».
- 🟢 **Задачи / заметки / воронка** — `POST /tasks`, `userComments`, статусы клиента (двигаются по API).
- 🟢 **Эквайринг Альфа = нативно + свой мерчант** (свои 1.4%, МК сверху 0%), оплата автопроводится в
  карточку по подписанному callback. **Ссылку на оплату можно отправить БЕЗ входа клиента в кабинет**
  (уникальная, сбрасываемая) → снимает то трение, из-за которого отвергли путь Fitbase.
- 🟢 **Вебхуки есть** (события, настраиваются сценарием в CRM; список — их статья 81).
- 🟢 **Данные в РФ** (ЦОД; см. ниже про расхождение) → 152-ФЗ residency; **согласие собирается прямо
  в онлайн-записи** (галочка + ссылка на наш текст, статья 660); есть «соглашение об обработке» (DPA).
- 🟢 **Модель «мама↔ребёнок»:** ребёнок = карточка (ученик), родитель = **признак** (доп. поле) в карточке
  ребёнка; **телефон — ответственного родителя**. Фича «Семья» — НЕ для нас (она для тех, кто сам ходит и
  делит абонемент). → по каждому ребёнку своя карточка/сделка.
- **Тариф:** API доступен на **Pro**; цена = тариф + число активных групп (дадут КП, если назвать число).

**Затыки (наша сторона / операционка):**
- 🔴 **Занятия не создаются наперёд** — их заводит сотрудник вручную на срок (или мы через `POST /lessons`).
  Иначе боту не во что записывать. → дисциплина «расписание заведено вперёд» либо наш код.
- **Нет поиска по chatId канала** + **`wazzupAssigns` мы сами не пишем** (только их интеграция/интерфейс)
  → держим **свой индекс chatId→ученик** (особенно MAX = 60%, там телефона в вебхуке нет).
- **Нет идемпотентности** → дедуп у нас; ссылка на оплату не сбрасывается → возможна двойная оплата (сверять).
- **Лимит 7 rps** → ошибка, при злоупотреблении блок IP; вебхук сам отключится, если наш приёмник 10 мин в таймаутах.
- ⚠️ **Слабое место — 1С:** готового коробочного коннектора нет (в отличие от Fitbase) — только экспорт
  из «Финансы» + интеграция по API / их платная доработка «Link». Уточнить, что реально нужно бухгалтеру.
- ⚠️ Хостинг назвали по-разному: продажи — **Netangels** (IP `5.181.253.117`), тех — **Ростелеком ЦОД Екб**.
  Оба РФ (residency ок), уточнить точную площадку для бумаг.

## 1С:Фитнес клуб — разбор

Источник: их API V3 (Apiary blueprint, `app.1c.fitness/fitnessapi`). Вопросы технарям заданы, **ждём ответ**.

**Сильное:**
- 🟢 **Нативный 1С** — закрывает бухгалтерию (главное слабое место Мой Класс).
- 🟢 **Встроенные согласия** (обработка ПД / оферта / политика) — прямо под 152-ФЗ.
- 🟢 Эквайринг **Альфа** (`use_bank=alfa`) + Сбер/Тинькофф/ЮKassa/СБП + `payment_link`; рекуррент/привязка карт.
- 🟢 `available_slots` (свободные места), `relatives` (родственники — чтение), `token-master` (родитель платит за ребёнка),
  лист ожидания, native статус брони `temporarily_reserved_need_payment` (возможно авто-холд — проверить).
- 🟢 Бронь ботом **достижима**, но через **«танец токенов»**: `GET /pass_token?phone&sign`
  (sign=SHA256 `phone:..;key:СЕКРЕТ`) → `reg_and_auth_client` (выдаёт `user_token`, создаёт клиента) →
  `client_to_class`. Серверно, без SMS клиента — **если у нас есть секретный ключ** (уточнить).

**Затыки:**
- 🔴 **API клиент-центричный** — каждое действие за клиента нужен `user_token` (а не чистый «admin + client_id»).
  Сложнее и хрупче Мой Класс; зависит от секретного ключа и жизненного цикла токена.
- 🔴 **Нет API задач**, **нет API заметок**, **нет поля воронки/этапа** (у лида только телефон/имя/коммент/utm + теги).
  → задачи = только алерт у нас; заметки → доп. поля/теги; воронку ведёт бот.
- ⚠️ Поиск клиента — через авторизационный флоу, не чистым админским GET.
- ⚠️ Привязка канала (chatId) — нет → наш индекс. Создание связи «родственник» по API — не подтверждено. Вебхуки/лимиты в спеке не описаны.

## MobiFitness — статус

- **MobiFitness ≠ 1С:Фитнес клуб** (в старых заметках были склеены — неверно).
- Это **своя платформа** (встроенная CRM + приложения), которая **может** работать и как
  **приложение поверх** внешнего бэкенда (1С:Фитнес или ClubIS).
- У их **собственной CRM** — **лучшая воронка продаж** (стадии, доска сделок, задачи, заметки) из всех.
- 🔴 **НО публичной документации API не найдено.** Как фундамент бота **нельзя проверить до договора**.
- **Ждём ответ:** «есть ли вообще открытое API для интеграции?» Если нет → как фундамент отпадает
  (воронка не спасёт). Брендированное **приложение** MobiFitness при этом можно поставить **поверх**
  1С:Фитнес как витрину — это отдельно от интеграции бота.
