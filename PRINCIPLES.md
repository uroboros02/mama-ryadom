# Принципы разработки проекта

> Как мы ведём работу над ботом. Файл живой — дополняй и правь.
> Контекст и архитектура самого бота — в [CLAUDE.md](CLAUDE.md).

## Порядок работы (4 роли)

Сначала — **архитектура по смыслу**: что делаем, в какой последовательности, что приоритетно. Код — только после.

1. **Архитектор** (на бумаге, до кода). Описываем словами: что строим, зачем, в каком порядке, что важнее. Сразу закладываем **безопасность**: где деньги/перс. данные/секреты, что может утечь, где зовём живого инженера. Каждую строчку документа ты понимаешь; непонятная строчка = стоп.
2. **Управляющий** (дробим). Большую задачу режем на маленькие, на каждую — короткое ТЗ словами. Весь проект разом не берём.
3. **Прораб** (пишем код). Код пишу я. На чувствительных местах — второе мнение (другая модель / живой инженер). «На слово» не верим.
4. **Приёмщик** (принимаешь ты). Проверяешь не код, а **поведение** по заранее описанным сценариям. Ведёт себя не так — не принято, откатываемся.

## Свод принципов

1. **Домен — твой, код — мой.** Бизнес-правило попадает в код, только если пришло от тебя.
2. **Сначала ТЗ словами, потом код.** Строчка, принятая «на веру», — будущая дыра.
3. **Маленькими кусками, один за раз.** Проверка после каждого.
4. **Проверяешь поведение, а не код** (сценарии на русском).
5. **Я объясняю каждое изменение простыми словами** (ритм — ниже).
6. **Простота по умолчанию, сложность — только по реальному поводу** (граница — ниже).
7. **Барьеры вокруг необратимого** (деньги, сообщения людям, запись в Fitbase) — через подтверждение человека, пока не обкатано.
8. **Git — кнопка «назад»:** каждое рабочее состояние = коммит.
9. **Логи — твои глаза:** бот записывает свои решения.
10. **Безопасность закладываем при проектировании, не переписываем потом.**
11. **Код пишем понятным для сторонних разработчиков** (ниже).
12. **LLM предлагает — код исполняет.** Claude (мозг) только *решает и предлагает* действие; любой side-effect с деньгами / скидками / этапами воронки / бронью / заметками / отправкой клиенту делает детерминированный код — он же **пере-проверяет** предложение по конфигу (возраст, пол↔направление, вместимость, скидка в списке) перед исполнением. Это **не** страховочное колесо (п.7, снимается с ростом доверия), а **постоянная граница**: даже на полной автономии LLM не «жмёт кнопки» напрямую. Зачем: модель может ошибиться/нагаллюцинировать — цена ошибки в деньгах/CRM высока, код-вентиль ловит это всегда. Отказ кода — **с конкретной причиной** (не «попробуй ещё»), чтобы бот исправился, а не повторил.

## Граница простоты и сложности (где буи)

- **Пол (ниже нельзя):** обработаны все случаи, которые **реально бывают** в бизнесе, + безопасность на границах (ввод клиента, деньги, перс. данные, причуды чужих систем — напр. повторные вебхуки Wazzup → не плодить дубли ответов). Проще этого = реальный баг/дыра.
- **Потолок (выше нельзя):** не обрабатываем то, чего не бывает; не строим «на вырост»; не плодим слои без причины сейчас. Сложнее этого = лишнее нагромождение.
- **Правило-решатель (твой инструмент):** на каждый непонятный кусок спрашиваешь — *«какой реальный случай в моём бизнесе это закрывает? покажи».* Конкретный случай (из домена или требование чужой системы) → оставляем. «Может пригодится / так принято / на будущее» → режем.
- **Кто ставит буи:** пол по бизнесу — ты (что реально бывает, твой домен). Пол по безопасности и весь потолок — слежу и подсвечиваю словами я, решаешь ты.

## Как я объясняю тебе код (ритм)

- **Перед задачей:** карта словами — что строим и как ляжет («на входе X, потом Y, этот блок отвечает за Z»).
- **После задачи:** короткий разбор по смыслу, блок за блоком (не построчно) + сценарии для проверки поведения.
- **По запросу:** на любой части — «разверни глубже».
- **В конце сессии:** короткая карта «что изменилось и где лежит».
- **Не делаем:** объяснять каждую строку на ходу (вода) и сваливать всё в конец (контроль теряется по дороге).

## Понятность кода для сторонних разработчиков

То, что делает код понятным «с улицы», — почти то же, что делает его понятным тебе по смыслу:

- **Имена по смыслу:** функция названа тем, что делает — `find_lead_by_phone`, а не `f1`. Прочитал имя — понял назначение.
- **Маленькие функции, одна задача каждая** (как абзац — одна мысль).
- **Скучно и стандартно > умно и хитро.**
- **Комментарий только там, где неочевидно ПОЧЕМУ** (не «что делает» — это видно из кода).
- **Предсказуемая структура файлов** + короткий документ-карта проекта.
- **Нет мёртвого кода и недоделок.**
- **Как проверяешь, не читая код:** спрашиваешь меня — *«открой этот файл глазами нового разработчика: имена говорящие? одна функция = одна задача? есть карта проекта?»*. Если карта понятна **тебе** по смыслу — будет понятна и инженеру.

## Защита кода и безопасность

### Где код может утечь

Код лежит в **3 местах**: git-репозиторий · хостинг/сервер · твой компьютер. Украсть можно только у того, у кого есть доступ к ним. Fitbase, Wazzup, Claude, клиент-пользователь код **не видят** — туда уходят только запросы/данные.

- **Безопасно давать** (код не утекает): подключение к Fitbase, Wazzup, Claude API.
- **Опасно давать** (утекает весь код): доступ к git-репозиторию, к хостингу/серверу. Только доверенным, репозиторий приватный, на хостинге 2FA.

### Главная защита от копирования

Против «украли код и запустили свою копию со своими ключами» техническая защита **ровно одна** — **не отдавать код**:

- **Хостинг = Вариант А:** бот крутится на нашем сервере, клиент подключает свои Fitbase/Wazzup, код к нему не попадает. Закончилось сотрудничество — рубильник у нас.
- Кому код всё же показываем (ревьюер) — держимся на **договоре/NDA**.

### Секреты (ключи)

Ключи = API-доступы к сервисам (Anthropic, Fitbase, Wazzup, Redis, Telegram).

- Держим в `.env` (локально) и в переменных окружения хостинга (продакшн) — **никогда в коде**. `.env` уже занесён в `.gitignore` → в git не попадёт.
- **Что защищает:** если код утечёт — ключи с ним НЕ утекут, и вор не потратит твои деньги (Claude) и не залезет в данные клиентов (Fitbase).
- **Что НЕ защищает:** само копирование кода (свои ключи вор подставит — см. «Главная защита» выше). Это разные линии обороны.

### Аудит у живого разработчика

Нужен для того, что не-программист не увидит: безопасность, платёжный флоу, хранение перс. данных (152-ФЗ), секреты/деплой.

- Ревью — **по законченным блокам** (F1, F2…), не разрозненными кусками: дыры живут в стыках, фрагмент без контекста проверить нельзя.
- Условия: **один доверенный разработчик + NDA**, доступ только на чтение, отозвать после.

### Системный промпт

Текст-инструкция, «мозг» бота и твоя методика продаж — **самая ценная и легче всего копируемая часть** (просто текст). Беречь, лишним людям не давать.
